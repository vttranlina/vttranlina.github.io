    One point is slightly stuck.

    Scenario:

    • Given Bob is a member of a Team Calendar and creates an event in it with ORGANIZER:mailto:bob@...
    • When an attendee changes PARTSTAT from their personal calendar
    • Then Sabre receives a REPLY sent to Bob, but the original event is not in Bob's personal calendar. It is in the delegated/canonical Team Calendar.

    The temporary fix adds a fallback only for REPLY: if the default lookup fails, it searches delegated Team Calendar nodes, It works, However, it looks not clean approach

  • comment on #424 — @vttranlina, 2026-07-10 15:08

    I don’t see any standard RFC field that directly supports our case: ORGANIZER is a real user like Bob, but the event object is stored under a team-calendar principal.

    The closest standard option is SENT-BY, for example: ORGANIZER;SENT-BY="mailto:team@example.com":mailto:bob@example.com or: ORGANIZER;SENT-BY="mailto:bob@example.com":mailto:team@example.com

    But neither fully matches our model: either replies still route to Bob, or the organizer semantic changes to the team identity.

    If we want to represent our current model explicitly, we may need a custom property, for example: X-OPENPAAS-TEAM-CALENDAR-ID:\<team-calendar-id\> and handle it specially during REPLY/event lookup.
