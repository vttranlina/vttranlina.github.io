  • comment on #978 — @chibenwa, 2026-07-21 16:03

    ## Analysis — GET /api/team-calendar/{id}

    Goal: expose a single team calendar over the front-facing REST API, symmetrical to GET /api/resources/{id} (ResourceRoute), mapping the fields we actually persist.

    ### What we persist for a team calendar

    The team_calendars Mongo collection (MongoDBTeamCalendarRepository) and the TeamCalendar model hold exactly:

    DB field model accessor
    _id id()
    domainId / domainName domain() (OpenPaaSId + Domain)
    name name()
    displayName displayName()
    creation creation()
    updated updated()

    So the DB gives us: id, domain, name, displayName, and the two timestamps. There is no description/type/icon/administrators/creator equivalent to a resource — those simply don't exist for a team calendar, so a strictly symmetrical response would carry a subset of the resource fields.

    Note that members are NOT in this DB — TeamCalendarMemberService.list(...) fetches them from DAV (fetchCalendarDetails with withRights), not from team_calendars. Since the issue asks for "mapping what we have in DB", I would keep members out of this endpoint (they already have their own webadmin route). If members are wanted here, that is a separate DAV round-trip and should be an explicit decision, not a DB mapping.

    ### Proposed response (symmetrical to ResourceResponseDTO)

    {
    "timestamps": {
        "creation": "2026-07-21T09:00:00.000Z",
        "updatedAt": "2026-07-21T09:00:00.000Z"
    },
    "_id": "{teamCalendarId}",
    "name": "sales-team",
    "displayName": "Sales Team",
    "domain": {
        "name": "open-paas.org",
        "_id": "{domainId}",
        "schemaVersion": 1,
        "administrators": [ ... ],
        "timestamps": { "creation": "..." },
        "hostnames": [ "open-paas.org" ],
        "injections": [],
        "company_name": "open-paas.org",
        "__v": 0
    },
    "__v": 0
}

    This reuses DomainRoute.ResponseDTO for the domain block and the timestamps { creation, updatedAt } shape exactly as ResourceRoute does, so the two endpoints stay consistent. Fields that resources have but team calendars do not (description, type, icon, administrators, creator, deleted) are simply omitted rather than sent as nulls. displayName is the one team-calendar-specific field added.

    ### Implementation sketch

    A new TeamCalendarRoute extends CalendarRoute in calendar-rest-api, mirroring ResourceRoute almost line for line:

    • endpoint() -> GET /api/team-calendar/{teamCalendarId} (or /api/team-calendars/{id} — see note below).
    • handleRequest: teamCalendarRepository.retrieve(new TeamCalendarId(param)) -> switchIfEmpty(NotFoundException) -> resolve+authorize the domain via the same retrieveAuthorizedDomainResponse(...) helper (OpenPaaSDomainDAO.retrieve + crossDomainAccessControl.denies(session, domain) + domainAdminDAO.listAdmins) -> serialize with the shared ObjectMapper.
    • Reuse of the existing cross-domain access control means a user cannot read a team calendar from a domain they may not access — same guarantee as ResourceRoute.
    • Register in RestApiModule (routes.addBinding().to(TeamCalendarRoute.class)).
    • TeamCalendarRepository is already injectable (used by TeamCalendarSearchProvider), so no new wiring beyond the route.

    Dependencies to inject: Authenticator, MetricFactory, TeamCalendarRepository, OpenPaaSDomainDAO, OpenPaaSDomainAdminDAO, CrossDomainAccessControl — identical set to ResourceRoute minus ResourceDAO.

    ### Points needing a decision

    1. Path naming — the issue title says /api/team-calendar/{id} (singular), but ResourceRoute uses the plural /api/resources/{id}. Recommend /api/team-calendars/{id} for consistency unless the frontend already expects the singular form.
    2. Members — confirm they are out of scope for this endpoint (DB-only), per the reasoning above.
    3. 404 vs cross-domain — as in ResourceRoute, both a missing calendar and a forbidden domain surface as NotFoundException (404), which avoids leaking existence across domains. Keeping that behaviour.

    ### Tests

    Add a TeamCalendarRouteTest under app/.../restapi/routes mirroring ResourceRouteTest: 200 with the exact JSON, 404 for unknown id, and a cross-domain access denial (404). Existing team-calendar test infra can seed a calendar to assert against.
