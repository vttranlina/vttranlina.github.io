TL;DR

  • Booking links moved forward significantly: webadmin CRUD merged, booker confirmation email merged, optional name/description merged, ICS now references booking links, and indexing/search work is open for follow-up.
  • Team Calendar architecture was clarified: side-service owns the Team Calendar entity, DAV permissions manage members, Team Calendars are event holders, not participants. First repository API/memory implementation merged; MongoDB implementation is open.
  • Address book/CardDAV webadmin API landed with list/create/delete/public visibility/delegation support and documented routes.
  • Test stability remains an active concern: several fixes landed, but WebsocketRouteTest and app-module flakiness still need monitoring.
  • SaaS-related plumbing improved: configurable calendar SaaS subscription queue names and auto-provisioning on settings were merged.

Highlights

Backend / Side Service / Sabre

Booking links

  • Webadmin CRUD for booking links was implemented and merged in pull/839, closing issues/837.
  • Booker confirmation email was added in pull/852, closing issues/828. Important security/product decision: avoid including unauthenticated user-provided input in the email body; wording was adjusted accordingly.
  • Optional booking-link name and description were added and merged in pull/861, closing issues/860. A MongoDB persistence gap was caught by tests and fixed before merge.
  • Optional auto-accept support was opened in pull/863 for issues/856; review signal was positive, but the PR was still open at the end of the reporting window.
  • Booking events now reference their booking link in generated ICS via pull/865, part of issues/864. Decision: use X-TWAKE-BOOKING-LINK and embed a stable internal identifier rather than the resettable public id.
  • Search/indexing follow-up is open in pull/869, tied to issues/867. Cleanup work is tracked separately in issues/868.
  • Booking event permalink/cancel/get design was clarified in issues/826: no update endpoint for now; frontend should cancel then create a new booking. Related route/token tasks were opened as issues/841, issues/842, issues/843, and issues/844.
  • JWT content discussion for booking event links concluded there was no major concern with including calendarId and ownerId, similar to the existing participation token approach in issues/826.

Address book / CardDAV webadmin

  • Address book admin webadmin interface was implemented and merged in pull/851, based on issues/801.
  • The API covers list, create, delete, public visibility, and delegation/invitee management. Documentation was added to docs/apis/webadmin.md.
  • Important implementation decision: deleting system address books should return 400 instead of leaking a 500; the route now checks address book type before attempting DAV deletion.
  • CardDAV semantics differ from CalDAV: public visibility and delegation are proxied via POST to Sabre; delegation sends the full desired sharee state, not a delta.

Team Calendar

  • Team Calendar design was substantially clarified in issues/515:
    • Side service owns a Team Calendar entity with id/name/etc.
    • Membership is managed through standard DAV permissions.
    • The organizer should be a write-enabled member; iTIP/IMIP replies use that user identity.
    • No Twake Mail team mailbox dependency and no virtual mail address for MVP.
    • Team Calendars are event holders, not participants; inviting Team Calendars is not supported for now.
    • People Search needs a Team Calendar provider.
  • Task split was created across side-service and Sabre work, including issues/829, issues/830, issues/831, issues/832, issues/833, issues/834, issues/835, plus Sabre-side issues esn-sabre/issues/391, esn-sabre/issues/392, esn-sabre/issues/393, and esn-sabre/issues/394.
  • Repository API and memory implementation merged in pull/857, closing issues/829.
  • MongoDB implementation was opened in pull/858 for issues/830, with review approval signal but still open in the report.
  • Rename handling was split out because updating display name likely requires renaming the underlying DAV collection and subscriptions: issues/859.

SaaS / AMQP / provisioning

  • Calendar SaaS subscription queue names are now configurable via pull/836, closing issues/824. Earlier overlapping work in pull/827 was closed.
  • SaaS settings now auto-provision registered users when needed via pull/838. Review clarified that this differs from TMail because Calendar needs a registered user, not just LDAP existence.
  • AMQP consumers now stop more gracefully on application shutdown via pull/847.

Review & Discussion Signals

  • Booking-link cleanup design has clear technical decisions from issues/864:
    • use X-TWAKE-BOOKING-LINK;
    • embed a stable internal booking-link id, not resettable public id;
    • expose booking-link criterion in event search;
    • cleanup must delete from CalDAV as source of truth;
    • cleanup must handle pagination because search results are capped.
  • Booking confirmation email review emphasized not reusing unauthenticated input. The booking title was excluded from the email in pull/852, with wording refined during review.
  • Team Calendar discussion converged on a cleaner model, but it implies Sabre work: separate principals/collections and iTIP awareness are tracked outside side-service. The unresolved implementation risk is the amount of Sabre/DAV work required.
  • Address book webadmin review caught documentation and test-quality issues in pull/851. The failing system-address-book delete test exposed a real error-mapping gap and led to a better proactive type check.
  • Test-stability reviews show that some fixes help but do not fully close the flakiness topic. In pull/866, the failing websocket test was due to an over-broad “no message” assertion rather than an actual import notification leak.
  • CodeScene comments were frequent. Most were low-signal approvals, but two useful signals were acted upon: duplicated tests in pull/851 and missing route-level tests for optional booking-link fields in pull/862.

Risks / Blockers

  • WebsocketRouteTest and broader app-module flakiness remain a risk. pull/866 improves message filtering semantics, but the underlying issues/818 was not clearly closed by the data in this reporting window.
  • Booking-link cleanup depends on search/indexing completion and must delete from CalDAV. Implementing cleanup against only OpenSearch would leave real events in calendars.
  • Team Calendar has cross-repo dependencies on Sabre changes; side-service work alone is not enough for the full feature.
  • No clear release blocker was visible in the data of this reporting window.
