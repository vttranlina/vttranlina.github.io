  • comment on #272 — @chibenwa, 2026-06-29 02:10

    Analysis — Refine CLASS = PRIVATE / CONFIDENTIAL

    I went through the current implementation to scope the work against @chibenwa proposal matrix.

    Target behaviour (from the proposal)

    viewer / class PUBLIC PRIVATE CONFIDENTIAL
    Owner read+write read+write read+write
    Public / regular read read free/busy (anonymised) nothing (filtered out)
    Public free-busy free/busy free/busy nothing
    Delegate (read) read read (full) free/busy
    Delegate (write) read+write read+write free/busy
    Participant (ATTENDEE) — — read (full)

    Two distinct mechanisms are required, as the ticket notes: PRIVATE/CONFIDENTIAL anonymises (replace details with a Busy placeholder) vs CONFIDENTIAL-to-public filters out (the object must not appear at all).

    Where the logic lives today

    The single decision point is Utils::isHiddenPrivateEvent() (lib/Utils/Utils.php:258), consumed by Utils::hidePrivateEventInfoForUser() (:192), which rewrites a hidden VEVENT down to SUMMARY: Busy + time/recurrence skeleton (:196-255).

    Current rule: if class is neither PRIVATE nor CONFIDENTIAL -> not hidden; otherwise hidden when getOwner() !== userPrincipal (getSourceOwner for subscriptions).

    So PRIVATE and CONFIDENTIAL are identical, and the only distinction is owner vs not-owner — there is no notion of delegate, participant, or filter-out. This matches the ticket two observations.

    Call sites that route through this logic (all need to honour the refined rules):

    • lib/JSON/Plugin.php:103 (iCal GET) and :215 (multiget/REPORT items)
    • lib/JSON/CalDAV/CalendarObjectHandler.php:345, 460, 526
    • lib/CalDAV/PrivateEventPlugin.php:199 — DAV-layer GET/PROPFIND sanitiser (uses an anonymous mock node exposing only getOwner())
    • lib/Publisher/CalDAV/EventRealTimePlugin.php:310 — pub/sub realtime payloads

    Gaps to close

    1. Delegate detection. Today a delegate sees only Busy on a PRIVATE event; the proposal wants delegates to see full details (PRIVATE) or free/busy (CONFIDENTIAL). The viewer relationship must be resolved from SharedCalendar::getInvites() / getShareAccess() -> ACCESS_READ / ACCESS_READWRITE / ACCESS_FREEBUSY (lib/DAV/Sharing/Plugin.php:11, lib/CalDAV/SharedCalendar.php). Note the mock node in PrivateEventPlugin only carries the owner — it would need the invite/share info too.

    2. PRIVATE != CONFIDENTIAL. Split the decision into an enum-like outcome (SHOW / ANONYMISE / FILTER_OUT) computed from (class, relationship): PRIVATE -> owner & delegates SHOW, everyone else ANONYMISE; CONFIDENTIAL -> owner & participants SHOW, delegates ANONYMISE, everyone else FILTER_OUT.

    3. Participant check (CONFIDENTIAL). Resolve whether the current principal is an ATTENDEE/ORGANIZER of the event. Requires principal->email mapping (see lib/Utils/Principal.php, currently minimal) and matching against ATTENDEE/ORGANIZER values.

    4. Filter-out path. hidePrivateEventInfoForUser() only rewrites; it never drops a VEVENT. For CONFIDENTIAL-to-public we need to (a) omit the object from collection/multiget responses and (b) make a direct GET return 404/403 in PrivateEventPlugin.

    5. Free/busy consistency. The matrix says CONFIDENTIAL shows nothing (not even busy) to public free-busy, but busy to delegates. lib/JSON/FreeBusyPlugin.php does not currently filter by CLASS, so it needs the same relationship-aware filtering — otherwise confidential time leaks via free-busy.

    Suggested approach

    1. Introduce a resolver returning the viewer relationship (owner / delegate-read / delegate-write / participant / other) given calendar node + principal, plus a classificationVisibility(class, relationship) returning SHOW/ANONYMISE/FILTER_OUT. Keep it in Utils so all five call sites share it.
    2. Refactor hidePrivateEventInfoForUser() to apply the outcome per-VEVENT (show / anonymise / remove), and have callers drop the whole object when every component filters out.
    3. Extend PrivateEventPlugin (enrich the mock node with invites; add 404/403 on filter-out for direct GET) and FreeBusyPlugin.
    4. Gate behind a flag (mirroring CALDAV_ORGANIZER_VALIDATION) so backend can ship CONFIDENTIAL ahead of the front, which targets public + private only — confidential can land disabled-by-default.

    Open questions

    • Delegation level for confidential: the proposal gives delegates free/busy only. Is there an intended confidential delegation level that grants full access (the wording without specific confidential access hints at one), or is participant-membership the only way to see confidential details?
    • Free-busy leak: confirm CONFIDENTIAL should be invisible (not even busy) to public free-busy — that is stricter than typical calendaring and worth an explicit decision.
    • Scope now: since the front targets public+private, should this PR fully implement confidential (filter-out + participant + free-busy) or only land the PRIVATE delegate behaviour and stub confidential behind the flag?

    Tests Unit: tests/Utils/UtilsTest.php (already covers hidePrivateEventInfoForUser) — add delegate/participant/confidential cases. Plus the DAV-layer PrivateEventPlugin and integration (IT) tests covering GET / multiget / free-busy across the matrix rows.

  • comment on #1093 — @chibenwa, 2026-06-29 17:08

    Analysis — x-openpaas-videoconference jCal type retro-compat

    Context: linagora/twake-calendar-integration-tests#280 flips the jCal value type of the x-openpaas-videoconference property from unknown to uri (e.g. ["x-openpaas-videoconference",{},"uri",""]). The backend now normalizes stored events to uri (PUT bodies sending unknown still return 201, and the subsequent GET returns uri). So the frontend must keep working whether the property comes back as unknown, uri, or anything else.

    1. Read / parse path — already format-agnostic ✅

    parseEventProperty (common/src/features/Events/utils/parseCalendarEvent.ts:232) destructures the jCal tuple as const [key, params, , value] = prop — the type element (3rd position) is discarded. The x-openpaas-videoconference handler (:170) only reads value. So unknown, uri, text, etc. all parse identically.

    This is already proven by the existing unit test eventUtils.test.ts:38, which feeds type text and passes. No change required to read both formats.

    2. Write path — hardcodes unknown, still functional ⚠️

    makeVevent (common/src/features/Events/utils/makeVevent.ts:29-33) always emits:

    ['x-openpaas-videoconference', {}, 'unknown', event.x_openpass_videoconference ?? null]

    This is the only 'unknown' literal in the events feature. The backend accepts it (integration tests PUT unknown → 201), so nothing breaks. To match the backend's canonical form it would be cleaner to emit 'uri' and an empty string instead of null, but it is not strictly required for correctness.

    3. Real retro-compat risk — series metadata diff in makeSeriesJCal ⚠️

    detectChangedMetadataFields (common/src/features/Events/transformers/makeSeriesJCal.ts:50-66) compares old vs new master properties with a full JSON.stringify of the whole tuple, including the type element. x-openpaas-videoconference is in METADATA_FIELDS (:19).

    After the backend change, the old master loaded from the server serializes as [...,"uri",""], while the new master built by makeVevent serializes as [...,"unknown",null]. The mismatch is twofold (type uri≠unknown, value ""≠null), so the field is flagged as "changed" on every series edit even when the video link was untouched — triggering needless propagation to all override instances and sequence bumps. (The "" vs null value mismatch likely pre-dates this type change; the type flip just widens the gap.) The existing test EventTransformers.test.tsx:357 only exercises the genuine-change case, so it would not catch this false positive.

    Recommendation

    • Read: nothing to do — type is already ignored. Optionally add a regression test parsing the property with type uri to lock the behavior in.
    • Write: align makeVevent to emit uri + empty string for the canonical form.
    • Series: make the metadata-diff in makeSeriesJCal robust to the type/empty-value representation — either compare only name/params/value (dropping the type element) or normalize both sides — so that a no-op video-link field stops being detected as a change. Add/adjust a test covering "master loaded as uri, edited via unknown writer → no spurious override propagation".

    Net: both formats are already supported on the read path; the concrete work is on the write side (canonical uri) and hardening the series metadata comparison.

  • Discussion Comments hiện crawl top-level comments; nested replies có thể được mở rộng sau nếu cần.
  • local_git repositories are collected from local commit history only.
