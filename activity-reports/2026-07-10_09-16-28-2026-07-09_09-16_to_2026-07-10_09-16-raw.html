  • comment on #943 — @chibenwa, 2026-07-09 18:54

    ## Analysis

    ### Where we stand today

    Booking link CRUD publishes nothing: BookingLinkCreateRoute, BookingLinkPatchRoute, BookingLinkDeleteRoute, BookingLinkResetPublicIdRoute and the WebAdmin BookingLinkUserRoutes all call BookingLinkDAO directly (no AMQP, no event bus, no listener). A second device — or a second tab — never learns that a link was created, edited, deactivated, deleted or had its publicId rotated.

    The WebSocket plumbing needed to fix this already exists and does not need to be invented. The closest precedent is the calendar list flow, which we can mirror almost line for line:

    Concern Calendar list (existing) Booking links (proposed)
    Event CalendarListChangedEvent (storage-api) BookingLinkChangedEvent (storage-api)
    Registration key UsernameRegistrationKey UsernameRegistrationKey
    Listener DefaultWebSocketNotificationListener same listener, extended
    Subscription auto-registered on connect (WebsocketRoute.registerDefaultSubscriptions, L279-292) auto-registered on connect
    Redis serialization CalendarEventSerializer.CalendarListChangedDTO BookingLinkChangedDTO
    Client message {"calendarList": {...}} {"bookingLinks": {...}}

    One important difference: calendar-list events are ingested from Sabre via RabbitMQ (CalendarListNotificationConsumer). Booking links are stored by us, in our own Mongo collection, so we dispatch on the event bus directly — no AMQP exchange, no consumer. The bus is Redis-backed (CalendarRedisEventBus), so a create issued on node A correctly reaches a WebSocket held on node B.

    ### Proposed API

    Subscription. None. Follow the calendarList precedent and register the booking-link listener in registerDefaultSubscriptions. Booking links are per-user, low-cardinality data that changes only on an explicit user action, so the traffic is negligible and an opt-in handshake would only add protocol surface. If you'd rather have opt-in, the alternative is a new prefix in ClientSubscribeRequest.toSubscriptionKey (WebsocketRoute.java:484) plus a BookingLinkSubscriptionKey — about twenty lines, and it can be added later without breaking the shape below.

    Server → client message, sent to every WebSocket connection of the link's owner:

    {"bookingLinks": {"created": ["3f2a...uuid"]}}
{"bookingLinks": {"updated": ["3f2a...uuid"]}}
{"bookingLinks": {"deleted": ["3f2a...uuid"]}}
{"bookingLinks": {"reset":   [{"from": "3f2a...uuid", "to": "9c1b...uuid"}]}}

    The created/updated/deleted arrays carry bare publicId strings and mirror CalendarListChangeMessage (DefaultWebSocketNotificationListener.java:67-99) exactly. The message is a notification, not a payload: the client reacts by refetching GET /api/booking-links. That keeps the message stable as BookingLinkDTO evolves, and avoids ever having to decide what a partial ValuePatch update should serialize to.

    reset is the one case that doesn't fit the shape, because resetPublicId destroys the old identifier and mints a new one — the client needs the correlation to update the row it is displaying rather than dropping and re-adding it. Modelling it as deleted:[old] + created:[new] would also work and would need no new shape, but it loses that correlation. I'd take the explicit reset entry.

    ### Where to dispatch

    BookingLinkDAO (storage-api/.../booking/BookingLinkDAO.java:26-40) is the only choke point through which all five mutations pass, from both the end-user API and WebAdmin. There is no service layer. So:

    Add an EventDispatchingBookingLinkDAO decorator in storage-api, wrapping the concrete DAO and holding an EventBus. Rebind in MemoryStorageModule.java:83 and MongoDBStorageModule.java:91. This gets WebAdmin notifications for free — an admin editing a user's link through /users/:username/booking-links will push to that user's open sockets, which is the behaviour we want and which a per-route dispatch would silently miss for two of the seven call sites.

    Each mutation has the data it needs without an extra read: insert and update already return the BookingLink, resetPublicId returns the new BookingLinkPublicId, and delete takes both Username and BookingLinkPublicId as arguments.

    The alternative — introduce a real BookingLinkService as the choke point — is more correct on layering grounds (a storage DAO emitting user-facing notifications is a smell) and would also let us delete the validateCalendarAccess logic currently copy-pasted across BookingLinkCreateRoute:150, BookingLinkPatchRoute and BookingLinkUserRoutes. It is a bigger change. Given CalendarListChangedEvent already lives in storage-api, the decorator is consistent with what is there.

    ### Things that will bite

    The Redis serializer is not optional. CalendarEventSerializer (calendar-redis/.../CalendarEventSerializer.java:44-193) has a closed @JsonSubTypes list and a default -\> null in toDTO. Forget to add BookingLinkChangedDTO and the event serializes to a SerializationResult.Failure — cross-node delivery breaks while single-node tests pass. This is the easiest way to ship a bug here.

    Dispatch failure must not fail the write. The DAO write is already committed by the time we dispatch. Log and swallow a bus failure rather than propagating it — a dropped notification should not turn a successful POST /api/booking-links into a 500. Clients recover by refetching on reconnect, which they must do anyway.

    Don't emit on a no-op delete. BookingLinkDAO.delete returns Mono\<Void\> whether or not anything was removed. BookingLinkDeleteRoute:63 checks existence first, but BookingLinkUserRoutes (WebAdmin) does not. Either guard in the decorator or accept spurious deleted messages.

    Both listeners share UsernameRegistrationKey. WebSocketNotificationListener (alarms) and DefaultWebSocketNotificationListener (calendar list) can both be registered under that key for the same connection. Since WebSocketNotificationListener.isHandling (L58-63) does not match BookingLinkChangedEvent, it will ignore it — nothing to change there, but worth an assertion in the tests.

    The public booking flow needs nothing. POST /api/booking-links/{id}/book creates a CalDAV event and never touches BookingLinkDAO (BookingLinkReservationService:76-120); the resulting change already reaches the owner's WebSocket as a CalendarChangeEvent via the calendar:event:created exchange. Same for the WebAdmin deleteEvents task. Neither should emit a bookingLinks message — the booking link itself did not change.

    publicId in the payload is safe. Events are keyed by the owner's UsernameRegistrationKey, so the rotated identifier only ever reaches the owner's own connections, and the owner has just received it in the HTTP response anyway.

    ### Scope

    • storage-api: BookingLinkChangedEvent (+ ChangeType {CREATED, UPDATED, DELETED, RESET}), EventDispatchingBookingLinkDAO
    • storage-api / storage-mongodb: rebind BookingLinkDAO to the decorator
    • calendar-redis: BookingLinkChangedDTO in CalendarEventSerializer, both switches
    • calendar-rest-api: extend DefaultWebSocketNotificationListener (isHandling, reactiveEvent, BookingLinkChangeMessage)
    • Tests: serializer round-trip; a WebsocketRouteTest case per change type; one asserting a WebAdmin-issued PATCH reaches the user's socket; one asserting a booking (/book) emits no bookingLinks message

    No new AMQP exchange, no queue, no consumer, no configuration flag.

    ## Analysis

    The fault-tolerant / convergent semantic requested here maps cleanly onto the existing machinery: a payload-free BookingLinkStateChangedEvent dispatched on the James EventBus keyed by UsernameRegistrationKey, delivered to the user's WebSocket by the listener that is already registered by default on every connection. No new subscription protocol, no delta.

    ### Current state

    Booking-link CRUD emits nothing today. I grepped storage/booking plus every write path for EventBus / dispatch / publish / amqp — zero hits. Mutations go straight to BookingLinkDAO.

    The four other WS notifications work like this (all in calendar-rest-api):

    Message Listener Registration key
    calendar sync token WebSocketNotificationListener CalendarURLRegistrationKey
    ICS/vCard import WebSocketNotificationListener Calendar/AddressBookURLRegistrationKey
    display alarm WebSocketNotificationListener UsernameRegistrationKey
    calendar list DefaultWebSocketNotificationListener UsernameRegistrationKey

    DefaultWebSocketNotificationListener is the interesting one: WebsocketRoute.registerCalendarListSubscription (line 285) registers it unconditionally at connection time, on UsernameRegistrationKey. Booking links are strictly per-owner and never shared, so that exact key is the right one — meaning we can piggyback on a registration that already exists, and clients get the message for free.

    ### Proposed API

    Server → client, no payload, flat, following the shape of the existing {"calendarListRegistered":true} ack:

    {"bookingLinkStateChanged": true}

    Emitted after any successful insert / update / delete / resetPublicId on a booking link owned by the connected user. The front re-fetches GET /api/booking-links on receipt. If a message is dropped, the next mutation re-triggers a full re-fetch; state converges. Nothing to reconcile, nothing to order.

    Client → server: nothing. No register entry, no toggle. Justification: the message is ~30 bytes, its rate is bounded by human CRUD, and the subscription already exists. Adding "register": ["bookingLinks"] would mean a new SubscriptionKey variant, a new prefix in ClientSubscribeRequest.toSubscriptionKey (which today only accepts /calendars/ and /addressbooks/ prefixes and throws otherwise), plus a client round-trip — for no benefit. Clients that don't know the key must ignore unknown top-level JSON keys, which is already required of them (alarms, imports, calendarList were each added over time).

    If you'd rather have symmetry with the alarm opt-in, the alternative is a {"enableBookingLinkNotification": true} toggle mirroring EnableAlarmDisplayNotificationRequest + AlarmSubscriptionKey. I'd skip it, but it is a two-record change if you want it.

    ### Where to emit

    The natural single choke point is a decorator over BookingLinkDAO, because there are two independent sets of write paths and a route-level hook would have to be repeated in both — and remembered by whoever adds the next one:

    • calendar-rest-api: BookingLinkCreateRoute, BookingLinkPatchRoute, BookingLinkDeleteRoute, BookingLinkResetPublicIdRoute
    • calendar-webadmin: BookingLinkUserRoutes (admin CRUD on behalf of an arbitrary :username)

    Every mutating BookingLinkDAO method already takes Username as its first parameter, so the decorator can dispatch to the owner's key even when an admin performs the write — which is exactly the behaviour you want for the webadmin path.

    public class EventBusBookingLinkDAO implements BookingLinkDAO {
    private final BookingLinkDAO delegate;
    private final EventBus eventBus;

    public Mono<BookingLink> insert(Username username, BookingLinkInsertRequest request) {
        return delegate.insert(username, request)
            .flatMap(link -> notify(username).thenReturn(link));
    }

    private Mono<Void> notify(Username username) {
        return Mono.from(eventBus.dispatch(
            BookingLinkStateChangedEvent.of(username),
            new UsernameRegistrationKey(username)));
    }
    // ... update / delete / resetPublicId identically; read methods delegate untouched
}

    Wiring caveat. BookingLinkDAO is bound in MemoryStorageModule:83 and MongoDBStorageModule:91, and neither module binds EventBus (it comes from RedisEventBusModule, or the in-VM fallback in TwakeCalendarMain.chooseCacheAndPubSub). Binding the decorator inside the storage modules would make them depend on an EventBus that isn't there in the DAO contract tests. Two ways out, pick one:

    1. Bind the decorator in a small app-level module (Modules.override(storageModule).with(...)), leaving the storage modules and BookingLinkDAOContract untouched. Least invasive.
    2. Introduce a BookingLinkManager in storage-api wrapping BookingLinkDAO + EventBus, and point the 4 REST routes and BookingLinkUserRoutes at it. More files, but gives you a home for the calendar-writability check that BookingLinkCreateRoute / BookingLinkPatchRoute currently duplicate (see #935).

    ### The thing that will be forgotten

    CalendarEventSerializer (calendar-redis, package org.apache.james.events) is a closed @JsonSubTypes list of four DTOs. A new event type that isn't added there serializes to SerializationResult.Failure and is silently dropped by CalendarRedisEventBus — so it will work perfectly in any test running on InVMEventBus and fail in production the moment Redis is enabled and the WS connection lands on another node. Add BookingLinkStateChangedDTO(eventId, username) to @JsonSubTypes, toDTO and fromDTO, and cover it with a serializer round-trip test.

    Event.isNoop() must return false, otherwise the bus drops the event before dispatch.

    ### Listener change

    Extend DefaultWebSocketNotificationListener rather than adding a class — it is already registered on UsernameRegistrationKey, and isHandling will keep filtering out the alarm events that share that key:

    public boolean isHandling(Event event) {
    return event instanceof CalendarListChangedEvent
        || event instanceof BookingLinkStateChangedEvent;
}

    emitting a new BookingLinkStateChangedMessage implements WebsocketRoute.WebsocketMessage.

    ### Semantics worth agreeing on up front

    • Spurious events are acceptable, missing ones are not. BookingLinkDAO.delete returns Mono\<Void\> and is idempotent, so deleting a non-existent link would emit an event with nothing having changed. Given the re-fetch semantic that is harmless, and I would rather have that than widen the DAO contract to Mono\<Boolean\> just to suppress it.
    • Bursts. resetPublicId, or an admin editing several links, produces one event each. The front should debounce (a couple hundred ms) before re-fetching. Worth writing into the doc so it is not rediscovered as a bug.
    • The public POST /api/booking-links/{id}/book endpoint does not mutate the link record — it only writes a calendar event via CalDAV. So it emits no booking-link event, and it should not: the resulting calendar change already reaches the owner through the existing CalendarChangeEvent path. Slot availability is derived from calendar events, not from the link.
    • BookingLinkEventDeletionTask (webadmin, ?action=deleteEvents) deletes generated calendar events, not the link. Same reasoning: no booking-link event.
    • InVMEventBus is single-node. With Redis disabled the event only reaches WS connections held by the same process. That limitation is pre-existing and identical for calendarList; just do not be surprised by it in a multi-node deployment that has not enabled Redis.

    ### Scope

    New: BookingLinkStateChangedEvent (storage-api), EventBusBookingLinkDAO or BookingLinkManager. Modified: CalendarEventSerializer, DefaultWebSocketNotificationListener, the storage/app module wiring, docs/apis/websocket.md (the Server → Client section plus the intro list of supported notifications). Tests: serializer round-trip; a WebsocketRouteTest case asserting the frame arrives on the default subscription without any register; an end-to-end test in the style of CalendarDavToWebsocketFlowIntegrationTest covering both a REST mutation and a webadmin mutation performed on behalf of the user.
