Highlights

Backend / Side Service / Sabre

  • Booking links

    • Booker confirmation email was merged in pull/852. A security/privacy decision was made not to include unauthenticated booker-provided booking title in the email body; wording was adjusted from “booking link request” to “booking request” (issues/828 comment, pull/852 comment).
    • Optional name and description for booking links were merged in pull/861. A MongoDB persistence bug was caught by tests and fixed by copying name/description into MongoDBBookingLinkDAO.insert (pull/861 comment).
    • Booking-link reference in generated event ICS was merged in pull/865. The branch was rebased after conflicts with pull/861 and reduced to the ICS-reference change only (pull/865 comment).
    • Booking-link search/indexing is in progress via issues/867 and pull/869. The agreed direction from issues/864: use X-TWAKE-BOOKING-LINK, embed a stable internal identifier rather than mutable public id, expose the criterion to end-user event search, and ensure cleanup deletes from CalDAV as source of truth (issues/864 decision comment).
    • Batch cleanup for booking links is now tracked separately in issues/868. It should build on the search/indexing work and must avoid deleting only from OpenSearch.

  • Address book WebAdmin / CardDAV

    • pull/851 added WebAdmin routes for user address book management and was merged.
    • The design covers list/create/delete, public visibility, and delegation/sharees; it proxies Sabre CardDAV operations and documents the routes in docs/apis/webadmin.md (issues/801 analysis, issues/801 updated analysis).
    • A test failure where deleting a system address book returned 500 instead of 400 was fixed by checking address book type before calling DAV delete (pull/851 fix comment).

  • Team calendar

    • Entity repository API and memory implementation were merged in pull/857.
    • MongoDB entity repository implementation is open and approved in pull/858.
    • A design concern was split out: renaming a team calendar likely requires renaming the underlying DAV collection and subscriptions, tracked in issues/859 from pull/858 inline comment.
    • Decision on invitations: team calendars should be treated as event holders, not participants; inviting team calendars is not supported for now (issues/515 comment).

  • SaaS user provisioning

    • On pull/838, the team clarified why Calendar auto-provisions registered users on settings while TMail checks LDAP-only user existence (pull/838 discussion, pull/838 answer).

Frontend

  • No dedicated frontend repository activity was included in this report.
  • Backend APIs are being prepared for frontend needs:
    • issues/837 comment states the related work is needed for a new frontend sprint.
    • Public agenda / booking event flow was decomposed into token signing, reservation route returning JWT, event get route, and cancel route in issues/826 comment, with new issues issues/841, issues/842, issues/843, and issues/844.
    • There was an open API-shape question on whether bookingLink needs to be returned in the booked event response or whether frontend can fetch it separately via slots API (issues/826 comment). Not enough signal in this window to conclude.

Review & Discussion Signals

  • Booking-link traceability decisions are mostly settled. From issues/864, the team agreed on X-TWAKE-BOOKING-LINK, stable internal id, end-user-visible search criterion, and CalDAV-based deletion for cleanup. Remaining implementation work is split into search/indexing pull/869 and cleanup issues/868.
  • Address book admin API design expanded during review. Initial list/create/delete scope was extended to public visibility and delegation. Review also forced documentation and test cleanup before merge in pull/851.
  • Test failures provided useful product signals. The WebSocket failure in pull/866 showed that calendarList notifications are independent of per-calendar registration; tests should assert specific message categories rather than “no message at all”.
  • Team calendar identity model needs care. The team clarified that team calendars are event holders, not participants, and identified DAV collection rename as a separate issue (issues/515 comment, issues/859).
  • Public agenda JWT security concern appears acceptable but not deeply reviewed. Including calendarId and ownerId in JWT was considered acceptable by analogy with ParticipationTokenSigner (issues/826 comment); no broader security review signal was present.

Risks / Blockers

  • Flaky tests are not fully resolved. issues/818 remains a risk; fork count change did not solve it, and pull/866 is still open.
  • Booking-link cleanup correctness risk. Cleanup must delete events from CalDAV and handle pagination/limits, not just remove OpenSearch documents. This is explicitly called out in issues/864 analysis.
  • Team calendar rename can create DAV inconsistency if not handled. Tracked by issues/859.
  • No P0 incident or release blocker is visible in the data for this reporting window.
