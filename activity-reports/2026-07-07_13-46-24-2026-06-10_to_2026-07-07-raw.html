  • comment on #389 — @chibenwa, 2026-06-25 20:29

    Analysis

    Context

    The codebase already has a clean precedent for optional PUT-time validation plugins: BinaryAttachmentPlugin (lib/CalDAV/BinaryAttachmentPlugin.php) hooks into beforeCreateFile/beforeWriteContent and CalendarObjectValidator (lib/CalDAV/Validation/CalendarObjectValidator.php) hooks into the calendarObjectChange event (via ESN\CalDAV\Plugin::validateCalendarObjectBeforeScheduling at priority 80).

    For ORGANIZER validation the calendarObjectChange event is the right hook: it fires for both creates and updates, the VCalendar is already parsed, and most importantly the $calendarPath parameter is provided which lets us resolve the effective calendar owner via $server->tree->getNodeForPath($calendarPath)->getOwner(). This is exactly how Schedule/Plugin::calendarObjectChange resolves the owner addresses.

    Proposed plugin: ESN\CalDAV\OrganizerValidationPlugin

    Hooks calendarObjectChange at priority < 80 (before scheduling). For each VEVENT component:

    1. If no ORGANIZER -> accept (per issue thread; see open question below)
    2. Resolve the ORGANIZER email to a principal via DAVACL plugin getPrincipalByUri
    3. Resolve the calendar owner principal via getOwner() on the calendar node
    4. Resolve the authenticated user principal via auth plugin getCurrentPrincipal()
    5. Accept if the organizer principal matches either the calendar owner or the authenticated user; otherwise throw Sabre\DAV\Exception\Forbidden

    This handles vttranlina's delegation case correctly: when Alice writes into calendars/bob/calendar1, getOwner() returns Bob's principal, so ORGANIZER=Bob is accepted. When Alice writes into her own calendars/alice/calendar1, the owner is Alice, so ORGANIZER=Bob is rejected.

    The plugin must skip ITIP deliveries (same guard as Schedule/Plugin::calendarObjectChange):

    if ($request->getMethod() === 'ITIP' || $request->getPath() === 'itip') {
    return;
}

    Open questions / decisions needed

    1. ATTENDEE without ORGANIZER

    RFC 5545 section 3.6.1 requires ORGANIZER when ATTENDEEs are present in a VEVENT/VTODO. The issue thread surfaced this but did not reach a firm decision (@chibenwa asked, @hungphan227 said "yes reject"). Recommend rejecting ATTENDEE without ORGANIZER (throw 403) since storing such an object will silently confuse ITIP routing. Needs explicit sign-off.

    2. Scope: VEVENT only, or VTODO/VJOURNAL too?

    The same RFC rules apply to VTODO. The issue mentions VEVENT but applying the same guard to VTODO from the start seems prudent.

    3. Should all VEVENT overrides carry the same ORGANIZER?

    RFC 5545 states ORGANIZER should be consistent across all components sharing a UID. The vobject library even has SameOrganizerForAllComponentsException. Recommend validating that all VEVENTs in the VCALENDAR carry the same ORGANIZER (or none). This is a stricter but correct check and avoids per-occurrence spoofing.

    4. Plugin enabled by default vs. opt-in?

    Like BinaryAttachmentPlugin, I would make this an optional plugin registered in the server bootstrap. The issue calls it "optional new plugin upon PUT" so opt-in seems intended.

    IT test scenarios

    • PUT with ORGANIZER = calendar owner -> accept
    • PUT with ORGANIZER = authenticated user (different from owner) -> accept (delegation)
    • PUT with arbitrary ORGANIZER (neither owner nor auth user) -> 403
    • PUT with no ORGANIZER -> accept
    • PUT with ATTENDEE but no ORGANIZER -> 403 (if we enforce RFC 5545 section 3.6.1)
    • PUT by Alice into Bob's delegated calendar with ORGANIZER=Bob -> accept
    • PUT by Alice into her own calendar with ORGANIZER=Bob -> 403
    • ITIP delivery with arbitrary ORGANIZER -> not validated (skipped)

    Review of CalendarObjectValidator

    I reviewed lib/CalDAV/Validation/CalendarObjectValidator.php + DateLikeValue.php, the wiring in lib/CalDAV/Plugin.php, the test suite, and cross-checked against RFC 5545 and the frontend fix in linagora/twake-calendar-frontend#1089.

    What it does today (and does correctly)

    The validator runs on calendarObjectChange (priority 80, before scheduling) and enforces, for VEVENT only:

    1. RRULE: COUNT/UNTIL mutual exclusion; UNTIL value-type and date-form rules (must be UTC when DTSTART is UTC or TZID, must stay FLOATING when DTSTART is FLOATING, must be DATE when DTSTART is DATE). This matches RFC 5545 §3.3.10 correctly.
    2. RECURRENCE-ID: matching recurring master exists (same UID with RRULE/RDATE), value-type + date-form match against the master's DTSTART, and uniqueness per UID.
    3. EXDATE: value-type + date-form match against the VEVENT's DTSTART (including multi-value EXDATE via getParts()).

    The date-form/value-type engine in DateLikeValue is solid and the test coverage of these three areas is good.

    ⚠️ Most important caveat: validation is opt-in

    validateCalendarObjectBeforeScheduling() returns immediately unless the request carries Prefer: handling=strict (Plugin.php:47). In the default (lenient) mode none of this validation runs — the protection only exists for clients that explicitly opt in.

    • Action: confirm the Twake frontend (PR #1089 flows: solo RSVP, single-occurrence delete) actually sends Prefer: handling=strict. If it doesn't, the validator gives zero protection for exactly the requests #1089 is about.

    Gaps worth deciding on

    1. It is purely syntactic — it does NOT catch the #1089 bug class. The validator checks that RECURRENCE-ID has the right value-type and timezone form, but never checks that the RECURRENCE-ID actually corresponds to a real instance of the recurrence set. The original #1089 bug produced a RECURRENCE-ID with the wrong time but the right form (browser-tz vs event-tz), which this validator would happily pass. Worth being explicit that it guards shape, not occurrence-alignment.

    2. Duplicate master is silently swallowed. mastersByUid() does $mastersByUid[uid] = $vevent; — two VEVENTs sharing a UID with no RECURRENCE-ID just overwrite each other (last wins). RFC 5545 requires exactly one master per UID in a resource; this should arguably be a validation failure rather than silently ignored.

    3. RDATE is not validated at all while EXDATE is. A master may recur via RDATE, but its value-type/form is never checked against DTSTART (and PERIOD-valued RDATE isn't considered). Asymmetry: a malformed/mismatched RDATE passes, an equivalent EXDATE fails.

    4. Stricter than RFC on TZID matching. For RECURRENCE-ID/EXDATE the code requires the exact form to match DTSTART, i.e. identical TZID. RFC 5545 only requires the same value type. Two valid-but-different zones (e.g. master TZID=Europe/Paris, EXDATE TZID=Europe/Berlin) would be rejected. The code comment shows this is deliberate (catching frontend bugs), which is fine for first-party traffic — but it can reject interoperable third-party clients. Worth a conscious decision + an explicit code comment that this is intentionally stricter than the RFC.

    5. VEVENT-only. VTODO (which can carry DTSTART/DUE/RRULE/RECURRENCE-ID/EXDATE) is not validated. Acceptable if VTODO isn't a concern, but should be stated.

    6. RRULE/EXDATE skipped when DTSTART is absent (!isset($vevent->DTSTART) → continue). A VEVENT with RRULE and no DTSTART is invalid per RFC; here it slips through this validator. In practice Sabre's own VEvent::validate() already requires DTSTART when there's no METHOD, so this is mostly covered upstream — worth confirming the strict-mode path still hits that.

    Minor

    • isValid() rejects leap-second times (...T086060Z), which RFC 5545 §3.3.5 technically permits. Negligible in practice.
    • select('RRULE') loops over multiple RRULEs per VEVENT (RFC says SHOULD NOT repeat) — handled gracefully, no action needed.

    Suggestion

    The core date/form engine is well-built and well-tested. If the goal is to actually prevent the #1089 family of regressions, the highest-value additions are: (a) verify/enforce that the strict header is sent, and (b) consider validating RECURRENCE-ID against the master's computed occurrence set (or at least RDATE), since form-matching alone won't catch a wrong-time override. Items 2 and 3 are cheap correctness wins.

    Analysis: why a filter-less calendar-query REPORT times out while export is fine

    Context

    The failing side is the reindex task in twake-calendar-side-service (linagora/twake-calendar-side-service#887). It enumerates every event of a calendar via:

    calDavClient.calendarQueryReportXml(user.username(), calendarURL, CalendarQuery.ofFilters())

    CalendarQuery.ofFilters() is an empty filter — i.e. a CalDAV calendar-query REPORT with no time-range and no prop-filter, asking for all objects + calendar-data. The MongoCursorNotFoundException (error 43) reported on the James side is a downstream symptom: the caller keeps its own Mongo cursor (iterating users/calendars) open while it waits for Sabre to answer; Sabre is so slow on a big calendar that the idle cursor exceeds its server-side timeout and gets reaped. The real problem to fix is here in esn-sabre: the filter-less REPORT is O(N) round-trips.

    Root cause: N+1 in the vendored calendarQueryReport

    The XML REPORT is handled by Sabre\CalDAV\Plugin::calendarQueryReport() (vendor/sabre/dav/lib/CalDAV/Plugin.php:622):

    $nodePaths = $node->calendarQuery($report->filters);   // 1 query -> list of URIs

foreach ($nodePaths as $path) {
    list($properties) =
        $this->server->getPropertiesForPath($requestUri . '/' . $path, $report->properties);  // <-- one PROPFIND *per event*
    ...
}

    calendarQuery() runs a single Mongo query and returns just the URIs, but then the plugin fetches calendar-data one event at a time via getPropertiesForPath(). Each iteration triggers a separate getCalendarObject() → a separate Mongo lookup. For the user in #887 that is ~26 500 individual Mongo round-trips + 26 500 VObject parses in a single request. That is what blows past every reasonable timeout.

    Why export is fast

    The ICS export path (Sabre\CalDAV\ICSExportPlugin::generateResponse, vendor/.../ICSExportPlugin.php:210) has no filter, so it takes the fast branch:

    $nodes = $this->server->getPropertiesForPath($path, [$calDataProp], 1);  // single Depth:1 fetch

    A single Depth-1 propfind resolves to Calendar::getChildren() → Backend\Mongo::getCalendarObjects($calendarId), i.e. one Mongo query that returns every object with its data. No per-event round-trip, no N+1. Same data, one query instead of N.

    Note esn-sabre already learned this lesson for the JSON report: CalendarObjectService::calendarQueryWithAllData() / getMultipleDAVItemsOptimized() fetch URI + calendardata + etag in a single query ({uri: {$in: [...]}}, see CalendarObjectDAO::findByCalendarIdAndUris). The XML calendar-query REPORT never got that treatment and still goes through the stock per-path Sabre loop.

    Proposed fix

    ESN\CalDAV\Plugin already extends Sabre\CalDAV\Plugin (lib/CalDAV/Plugin.php:19), so we can override calendarQueryReport to batch the data retrieval instead of looping one path at a time:

    • Minimal / low-risk: replace the per-path getPropertiesForPath() loop with a single getPropertiesForMultiplePaths($nodePaths, $report->properties) (exactly what ICSExportPlugin does). This turns N getCalendarObject calls into one getMultipleCalendarObjects ($in query). Behaviour, filtering and output are unchanged.
    • Best: for the common empty-filter case (no post-filter needed — see buildQueryFromFilters, requirePostFilter=false), route straight to the existing calendarQueryWithAllData() so URIs + data come back in a single query and each object is parsed at most once. Fall back to the standard path only when a real post-filter is present.

    Either option removes the N+1 and brings the REPORT in line with export's single-query cost, which should make the reindex complete well within the caller's cursor timeout. On the twake side it would also be worth raising/refreshing the enumeration cursor (noCursorTimeout or smaller batches) as defence in depth, but that's secondary once the REPORT itself is fast.
