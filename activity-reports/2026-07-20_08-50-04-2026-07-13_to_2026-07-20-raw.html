  • comment on #959 — @chibenwa, 2026-07-17 01:39

    Expanded extraAttendees into a tree as suggested, in d10a2bca.

    Wire format — a single and of participant leaves, which expresses exactly today's semantic (invite all of them):

    { "extraAttendees": { "and": [{ "participant": "id1" }, { "participant": "id2" }] } }

    Any other shape — other node types, nested and, extra fields on a leaf — is rejected with a 400 for now. So the payload you sketched for later (or under and) is a purely additive change: a new record in the sealed ExtraAttendeeNode plus the branch in the parser.

    Model — ExtraAttendeeNode is a sealed interface (And, Participant) able to hold arbitrary trees; ExtraAttendees wraps the root and surfaces the flat list via participants(). The resolvers are untouched: they still take a List\<OpenPaaSId\>, callers just pass extraAttendees().participants().

    DB — the Mongo field is now a nested document mirroring the tree ({and: [{participant: "id1"}]}), and both serialization and deserialization walk it recursively, so arbitrary trees already round-trip. No migration: the flat-array format only ever existed on this branch.

    No behaviour change otherwise — slot intersection, booked-event attendees, the 20-entry cap, dedup and the owner/unknown-id validation all work as before.

    Two things worth flagging beyond the requested change:

    1. A regression from my previous review fix. BookingLinkExtraAttendeeResolverTest.validateShouldRejectUnknownUserWhenOwnerIsNotRegistered was failing on 54104f25. Dropping the cache() the way it was suggested — retrieve(owner).flatMap(ownerId -\> ...) — nests the attendee check under the owner lookup, so an owner who is not a registered user makes the Mono empty and unknown extra attendees stop being rejected. Fixed by resolving the extra attendees first and checking the owner against them, which still subscribes to the owner exactly once and needs no cache().

    2. A Jackson trap this restructuring walked into. Typed as Optional\<JsonNode\>, an absent field binds to Optional.of(NullNode), not Optional.empty — unlike Optional\<List\<String\>\>. So every create/patch omitting extraAttendees briefly returned 400. Absent and explicit-null are now normalized in one place, and BookingLinkExtraAttendeeUtilTest pins both. This also let the duplicated patch parsing in BookingLinkPatchRoute and BookingLinkUserRoutes collapse into BookingLinkExtraAttendeeUtil.parsePatch.

    Verification: Docker was available this time, unlike in my previous session, so everything ran locally and passes: BookingLinkExtraAttendeeUtilTest (24, new), MongoDBBookingLinkDAOTest (42, real Mongo), BookingLinkUserRoutesTest (34), BookingLinkExtraAttendeeResolverTest (7), plus the app suites BookingLinkCreateRouteTest (43), BookingLinkPatchRouteTest (42), BookingLinkGetRouteTest/BookingLinkListRouteTest (18), BookingLinkSlotsRouteTest (25) and BookingLinkReservationRouteTest (35).

  • comment on #426 — @chibenwa, 2026-07-16 11:09

    Thanks @vttranlina — good point, and this is really the core design tension here.

    Weighing the two options:

    Direct MongoDB write (current PR)

    • No new runtime dependency: Sabre already owns a Mongo connection and already reads the users collection for principal resolution, so provisioning writes into a store it already talks to.
    • The document format is deliberately aligned with MongoDBOpenPaaSUserDAO (same firstname / lastname / firstnames / password / email / domains[].domain_id / accounts[].type,emails shape) so both services share the same schema.
    • Downside: it couples Sabre to the side-service's storage schema. If that schema evolves, both writers must stay in sync.

    Call the side-service webadmin/API (your alternative)

    • More stable: the side-service stays the single owner of its user model, so schema changes do not leak into Sabre.
    • Downside: a synchronous cross-service call on the auth hot path — extra latency, a new failure mode (side-service down => auth fails), plus config/credentials for that endpoint. It also inverts the current dependency direction (today Sabre does not call the side-service).

    I have kept the direct-write approach in this PR because it introduces no new cross-service dependency on the authentication path, but the schema-coupling concern you raise is real. Since this is an architectural call rather than a bug, I would rather not rewrite the provisioning path unilaterally — happy to switch to the webadmin/API route if the team prefers the stability tradeoff. Let me know which direction you want and I will adjust.
