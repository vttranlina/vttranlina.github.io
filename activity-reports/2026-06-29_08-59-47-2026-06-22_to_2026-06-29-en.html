TL;DR

Backend / Side Service / Sabre

Booking links

  • Side-service merged several booking-link features:

    • Booker confirmation email in pull/852, with explicit avoidance of unauthenticated user-provided content in email text.
    • Optional name/description in pull/861; a Mongo persistence bug was caught and fixed during review.

  • Technical decisions for cleanup/search:
    • Use X-TWAKE-BOOKING-LINK, not X-OPENPAAS-BOOKING-LINK.

  • Booking-event permalink/cancel/get JWT flow is scoped in issues/826, with follow-up issues issues/841, issues/842, issues/843, and issues/844.

Address book / CardDAV webadmin

  • Address book WebAdmin interface was implemented and merged in pull/851, covering list/create/delete, public visibility, and invitee/sharee management.
  • Review uncovered a real test/behavior issue: deleting a system address book returned 500 instead of the expected 400. The fix proactively checks address book type before DAV deletion. Source: pull/851 comment.
  • The design explicitly notes CardDAV delegation differs from CalDAV: CardDAV dav:share-resource sends the full desired state, not a delta. Source: issues/801 comment.

Team Calendar

  • Team Calendar architecture was clarified in issues/515:

    • DAV permissions manage members.
    • Sabre should support a separate team-calendar principal model.
    • ORGANIZER can be a write-enabled member of the Team Calendar.
    • No initial support for inviting Team Calendars as participants; they are event holders.
  • Task split was created across side-service and Sabre:
    • Side service: issues/829, issues/830, issues/831, issues/832, issues/833, issues/834, issues/835.
    • Sabre: issues/391, issues/392, issues/393, issues/394.
  • Entity API/memory implementation was merged in pull/857. Mongo implementation is open in pull/858.
  • Rename of underlying DAV collection/subscriptions was identified as split work in issues/859, from pull/858 review comment.

Sabre validation

  • Sabre ORGANIZER validation was implemented and merged in pull/395, with integration coverage in pull/278.
  • Key decisions for ORGANIZER validation:
    • No ORGANIZER is allowed.
    • ATTENDEE without ORGANIZER should be rejected.
    • Scope is VEVENT only.
    • Plugin is opt-in initially.
    • Source: issues/389 comment.
  • Sabre also merged pull/399, requiring DTSTART for RRULE/EXDATE and documenting strict TZID matching.
  • A validation review noted that strict validation only applies when clients send Prefer: handling=strict, and that recurrence-id form validation does not prove occurrence alignment. Source: issues/398 comment.

Frontend

  • Major recurring-event regression fixed in pull/1089:
    • Root cause: solo RSVP and recurring instance updates could regenerate VEVENT overrides with timezone-shifted or duplicate RECURRENCE-ID.
    • Fixes included in-place jCal PARTSTAT patching, instant-aware recurrence matching, preserving stored RECURRENCE-ID tuples, EXDATE TZID-form emission, and regression tests.
    • Manual validation covered timezone changes, overrides, partial acceptance, and EXDATE. Source: pull/1089 comment.
  • SaaS Visio URL generation was clarified and merged in pull/1092:
    • VIDEO_CONFERENCE_BASE_URL is now an admin-controlled template.
    • Supported expressions include {localpart}, {workplaceFqdn}, {workplaceFqdn.localpart}, and {workplaceFqdn.domain}.
    • localpart must come from userData.email, not sub, because sub is an opaque id. Source: pull/1092 review comment.
  • Frontend alarm scheduling with multiple alarms and attendee support is open in pull/1091. It received functional validation, but review comments flagged areas around personal/global alarm merging, empty attendee placeholders, multi-select empty option behavior, and Valarms consistency.
  • User API refactor to DAO/transformer structure was merged in pull/1062. Review signals focused on avoiding raw _id/id ambiguity, pushing parsing into domain types, and ensuring stored class-like objects are rehydrated correctly.
  • Repetition serialization work remains open in pull/1073, with architectural feedback to make event.repetition reliably a RepetitionObject and move normalization/serialization behavior into the model.
  • userOrganiser.asJcal() work in pull/1075 was approved after feedback that asJcal() must always be available, including after Redux/store serialization paths.
  • Public booking frontend scope was broken down through issues issues/1052 to issues/1061, plus follow-ups for booking creation/edit/permalink in issues/1076, issues/1080, issues/1081, and issues/1082.

Review & Discussion Signals

  • Team Calendar model shifted toward a cleaner separated principal/entity model. Earlier “no Sabre work” thinking changed; current direction favors /principal/team-calendar and separate Sabre support for maintainability. Source: issues/515 comment.
  • Team Calendar invitation semantics were narrowed: team calendars are event holders, not participants, at least initially. Source: issues/515 comment.
  • Booking-link cleanup has three explicit design decisions to preserve: stable internal id, end-user-visible search criterion, and CalDAV deletion. Source: issues/864 comment.
  • Frontend recurring-event work showed the value of strict Sabre validation: frontend bugs around RECURRENCE-ID and EXDATE were caught by Sabre CalendarObjectValidator errors during manual testing in pull/1089.
  • Several frontend review threads converge on the same architectural theme: class-like domain models (RepetitionObject, userOrganiser, Resource, Valarms) must be reliably constructed/rehydrated before callers depend on methods like asJcal().
  • CodeRabbit/CodeScene generated many low-level comments; the useful signals were mostly correctness/data-integrity issues, not style nits. Several bot claims were explicitly withdrawn after project-specific clarification, e.g. MUI picker class names and event.form.notifications. translation key.

Risks / Blockers

  • pull/1091 is still open and touches complex alarm scheduling semantics. Main risk: accidental duplication/loss of global vs personal alarms, blank attendee identities, or invalid multi-select state.
  • pull/1073 remains architecturally sensitive: type/model coupling and casts were questioned, and the desired direction is to push repetition normalization/serialization into the data model.
  • Side-service flaky WebSocket tests are not fully closed. pull/866 addresses one failure mode, but issues/818 still has “not certain fully solved” signal.
  • Booking-link cleanup is not implemented yet; search/indexing is only part of the prerequisite chain. Cleanup must avoid relying on a single capped search page and must delete from CalDAV.
  • Sabre CalendarObjectValidator strictness only protects requests using Prefer: handling=strict; confirm frontend coverage where the validator is expected to catch regressions.
