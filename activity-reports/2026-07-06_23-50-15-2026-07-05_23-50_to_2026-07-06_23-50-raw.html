  • comment on #914 — @chibenwa, 2026-07-06 16:23

    Analysis

    Short answer: yes, the slot is validated server-side — a hand-crafted curl cannot book an arbitrary date. But there are a couple of gaps worth deciding on.

    Where validation happens

    POST /api/booking-links/{id}/book → BookingLinkReservationService.book() calls validateSlotAvailability() before creating the event (BookingLinkReservationService.java:79-95):

    return bookingLinkSlotsService.computeSlots(bookingLink, queryStart, queryEnd)
    .filter(slots -> slots.contains(new AvailabilitySlot(startUtc, bookingLink.duration())))
    .switchIfEmpty(Mono.error(() -> BookingLinkReservationException.notAvailable(...)))

    The requested startUtc must be exactly equal to one of the server-computed AvailabilitySlots (same start, same duration). Those slots are recomputed on the server from the same source as the public /slots endpoint (BookingLinkSlotsService.computeSlots): availability rules minus busy intervals from CalDAV (AvailableSlotsCalculator.Default). An unavailable/misaligned slot yields 422 UNPROCESSABLE_ENTITY. So:

    • Times that do not fall on the availability grid are rejected — slots are generated minute-aligned, stepping by duration from the availability-range start (AvailableSlotsCalculator.java:99-108). A startUtc with stray seconds or off-grid minutes will not match.
    • Times overlapping an existing busy event are rejected (unavailableTimeRanges removed in computeAvailabilityRanges).
    • Times outside the configured weekly/fixed availability rules are rejected.

    The fear of full "openbar" registration is therefore not the case for a link that has availability rules configured.

    Gaps worth considering

    1. No lower time bound / no minimum-notice check. Nothing checks that startUtc is in the future. Weekly rules are recurring, and the validation window is derived purely from the user-supplied startUtc (queryStart = startUtc - duration, queryEnd = startUtc + duration), independent of "now". A curl caller can therefore book a slot in the past as long as it matches a weekly availability rule (e.g. last Tuesday 10:00). The Clock injected into the service is only used for the ICS DTSTAMP, not for validation. Note the public /slots endpoint (BookingLinkSlotsRoute) also never clamps from to now, so the UI could equally query past ranges. A configurable "minimum booking notice" / future-only guard would close this.

    2. Empty availability rules means effectively open. When bookingLink.availabilityRules() is empty, BookingLinkSlotsService.toComputeSlotsRequest falls back to a FixedAvailabilityRule spanning the entire query window (BookingLinkSlotsService.java:80-85). Combined with gap 1, that means any minute-aligned, non-busy instant becomes bookable at any date. If a link is never meant to exist without rules, that invariant should be enforced at creation; otherwise this is the real "openbar" scenario.

    3. No throttling on the public endpoint. Independent of slot validity, /book is unauthenticated (PublicRoute); nothing rate-limits creation. Even valid-slot bookings can be spammed (one event per free slot). Rate-limiting / captcha is a separate hardening item if abuse is a concern.

    Suggestion

    The core slot-availability validation is present and correct. I would recommend: (a) add a future/minimum-notice check in validateSlotAvailability using the injected Clock, and (b) decide whether a booking link may exist without availability rules, enforcing it at creation if not.

  • comment on #919 — @chibenwa, 2026-07-06 20:06

    Analysis

    Current behaviour

    The booking endpoint POST /api/booking-links/{publicId}/book is served by BookingLinkReservationRoute, which extends PublicRoute — i.e. it is fully unauthenticated (calendar-rest-api/.../routes/PublicRoute.java). The booker's identity is taken verbatim from the request body:

    { "creator": { "name": "...", "email": "..." }, ... }

    ReservationRequestDTO only validates that creator.email is non-blank and well-formed; it never checks who is actually sending the request. That creator is then turned into the event's ORGANIZER/ATTENDEE with the supplied CN and mailto: (BookingLinkEventIcsBuilder lines 124, 159-184), and stored as X-PUBLICLY-CREATOR. The ICS is imported directly into the organizer's calendar (BookingLinkReservationService.createBooking).

    Consequence: anyone can book a slot while declaring creator.email = alice@company.com (a real platform user). The organizer then sees a booking that appears to originate from a trusted colleague — exactly the spoofing concern raised here and in linagora/twake-calendar-frontend#1109.

    Why "when possible"

    We cannot require auth unconditionally: external/anonymous booking is the whole point of a booking link (the frontend excal form). So the check has to be conditional — enforce identity only when the declared identity belongs to a registered user, and leave genuinely external bookers untouched. Hence "when possible".

    Proposed approach

    1. Make the route optionally authenticated. Today the split is binary: PublicRoute (no auth) vs CalendarRoute (authenticator.authenticate(req), which hard-fails on missing/invalid credentials). We need a third mode: attempt authentication, tolerate its absence, but still reject genuinely invalid credentials. The frontend (#1109) will forward OIDC creds when present and fall back to the anonymous excal form otherwise, so the backend must accept both a valid session and no session.

    2. Detect a registered identity. Resolve creator.email via OpenPaaSUserDAO.retrieve(Username.of(email)). Non-empty means the declared creator is a local user.

    3. Enforce the match. If the creator is a registered user:

      • require an authenticated session, and
      • require session.getUser() to match creator.email (case-insensitive),
      • otherwise reject with 401 (no/invalid creds) or 403 (authenticated as someone else).

      If the creator is not a registered user, keep the current anonymous flow unchanged.

    4. Trust the platform identity, not the body. When authenticated, derive the creator's display name (and ideally email) from the resolved OpenPaaSUser.fullName() rather than the client-supplied name, as the issue states ("the registered user display name is then used"). This also prevents a valid user from setting a misleading CN.

    Points to settle before implementing

    • Identity resolution must match auth's view of identity. Detection in step 2 must use the same username/alias resolution as the authenticator (see UserNameResolver), otherwise an attacker could dodge the check by using an alias of a registered user that retrieve() does not resolve but which is still a real inbox. This is the main correctness risk.
    • Additional attendees. additional_attendees are still fully self-declared. Impersonating them is lower risk (equivalent to inviting someone to any event), so I would scope this change to creator only — but worth noting explicitly.
    • Invalid-credentials policy. Decide whether presenting invalid/expired credentials on an anonymous booking should fail hard or degrade to anonymous. Failing hard is safer and avoids a silent downgrade.
    • CORS / preflight are already handled by the route builder; no change needed there.
    • Tests. Add cases to BookingLinkReservationRouteTest: registered creator without auth => rejected; registered creator with mismatched auth => rejected; registered creator with matching auth => created and display name overridden; external creator => unchanged.

    Recommendation

    Worth doing — it closes a real spoofing hole at low cost and improves UX for logged-in users. The only non-trivial part is wiring optional authentication into a route that is currently public, and making sure registered-user detection uses the exact same identity/alias resolution as authentication. I would implement it as: optional auth in the route + a check in BookingLinkReservationService.book(...) that takes an Optional<Username> authenticatedUser and enforces the rules above.

  • comment on #1109 — @chibenwa, 2026-07-06 20:03

    Analysis

    I agree with the proposal — it is a genuine UX + security improvement. A few remarks on how it maps onto the current code, and one caveat that matters.

    Where we stand today

    • We ship two separate SPAs: apps/private (authenticated, OIDC/PKCE via common/src/features/User/oidcAuth.ts + useInitializeApp) and apps/public (anonymous). The "excal" flows live in the public app: apps/public/src/App.tsx routes /excal to EventPreviewPage and /booking/:bookingLinkPublicId to BookingPage.
    • BookingPage (apps/public/src/features/booking/BookingPage.tsx) is currently pure scaffolding (returns null, only fetches slots). So the anonymous booking UX itself is not built yet — good, most of this is greenfield.
    • The booking DAOs are split: the public BookingDao exposes fetchBookingSlots + createBooking (no auth, identity passed in the body as creator: { email, name }), while the private BookingDao only does owner-side CRUD on links. There is no authenticated "book a slot" call today.

    The design fits, with these pieces to add

    1. A private booking page mirroring the public one, either as a new route in apps/private or by making the private app own /booking/:bookingLinkPublicId.
    2. Soft auth detection. Today useInitializeApp forces a redirect to SSO on load. A booking landing page must instead detect an existing session (the tokenSet/userData in sessionStorage) without auto-redirecting, so an anonymous visitor still gets the "log in / continue on excal" choice. This is a deliberate departure from the current always-redirect behaviour and should be scoped to the booking route only.
    3. An authenticated booking endpoint / DAO. For the OIDC path we need a createBooking variant that sends the auth token and lets the server derive creator from the token.

    The one caveat: the security win must be enforced server-side

    "Prefill name/email, not editable in booking" is a UX affordance, not a security boundary — a client can always forge the request body. The anti-impersonation guarantee ("easy to register as a known platform user") only materialises if the backend, on the authenticated endpoint, sets the attendee identity from the OIDC token and ignores/validates any creator in the body. Otherwise we have only hidden the field, not closed the hole. So this issue has a backend dependency (calendar-side) that should be tracked explicitly. The current unauthenticated createBooking (public) trusting body creator must remain the anonymous-only path.

    Cross-app plumbing

    Since private and public are distinct deployments, redirecting between them ("offer to log in" versus "redirect to excal") needs a configured base URL for the other app. window.d.ts has no such config yet (we have MAIL_SPA_URL, LANDING_PAGE_URL, ...), so add one (e.g. a public/excal base URL) rather than hard-coding.

    "Default landing page for excal"

    When we build the owner-side UI that generates/shares a booking link, the shared URL should point at the private booking route by default; anonymous visitors are then offered login-or-continue, and "continue" bounces them to the existing public /booking/:id. Deciding this now avoids re-issuing already-shared links later.

    Suggested split

    • BE: authenticated booking endpoint deriving attendee identity from token (blocker for the security claim).
    • FE step 1: private booking page + soft session detection (no forced SSO redirect).
    • FE step 2: authenticated createBooking DAO + prefilled/read-only identity.
    • FE step 3: anonymous fallback (login vs. redirect to excal) + cross-app base-URL config.
    • FE step 4: link generation defaults to the private route.

    Overall: +1. The only thing I would flag as non-negotiable is the server-side identity enforcement — without it the "not really secure" problem is not actually fixed.
