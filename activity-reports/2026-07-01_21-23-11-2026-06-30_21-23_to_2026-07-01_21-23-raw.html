  • comment on #887 — @chibenwa, 2026-07-01 14:50

    Analysis

    Root cause

    The reindex-calendar-events task iterates users and resources by streaming an open MongoDB cursor while doing very slow per-item work on it, and the cursor is reaped server-side before the next getMore.

    In CalendarEventsReindexService.reindex(...) the source is:

    Flux.concat(
    userDAO.list().concatMap(user -> collectEvents(...)),          // one open cursor
    resourceDAO.findAll().concatMap(resource -> collectEvents(...))) // another open cursor

    Both MongoDBOpenPaaSUserDAO.list() and MongoDBResourceDAO.findAll() are plain streamed cursors with no batching/timeout tuning:

    • MongoDBOpenPaaSUserDAO.java:158-160 -> Flux.from(database.getCollection(COLLECTION).find())
    • MongoDBResourceDAO.java:96-97 -> Flux.from(collection.find()).map(...)

    The reactive driver (com.mongodb.reactivestreams) reads this cursor batch by batch: it fetches one batch, emits it downstream, and only issues the next getMore once that batch is drained. Draining a batch here is extremely expensive and rate-limited:

    • default calendarsConcurrency = 1 (CalendarEventsReindexTask.java:43) -> users and their calendars are processed sequentially via concatMap;
    • for each item we deleteAll from the search index, hit CalDAV (calendarQueryReportXml), parse ICS, and reindex;
    • the whole indexing pipeline is throttled to eventsPerSecond (ReactorUtils.throttle(), lines 142-145).

    With 26540 events and this sequential/throttled processing, the time spent between two consecutive getMore calls on the user cursor easily exceeds MongoDB's default idle-cursor timeout (cursorTimeoutMillis, 10 minutes). The server then reaps the cursor, and the next getMore fails - exactly the stack trace reported:

    MongoCursorNotFoundException: cursor id ... not found (code 43, CursorNotFound)
  at ...AsyncCommandBatchCursor.lambda$getMoreLoop$2

    This propagates up past the per-user/per-resource onErrorResume handlers (those wrap the inner collectEvents monos, not the outer source Flux), reaches the top-level .onErrorResume at lines 155-158, which logs "Task reindex is incomplete" and returns Task.Result.PARTIAL. That is why the task ends failed/partial even though failedEventCount = 0: the failure is the source cursor dying, not any individual event.

    Fix

    Decouple the slow downstream processing from the live driver cursor. The simplest, robust option is to drain the user/resource listing into memory first, closing the cursor quickly, then process:

    userDAO.list()
    .collectList()
    .flatMapMany(Flux::fromIterable)
    .concatMap(user -> collectEvents(context, user, runningOptions.calendarsConcurrency()))

    (and the same for resourceDAO.findAll()). OpenPaaSUser/Resource are lightweight records and their cardinality is bounded (thousands, not the 26k events), so buffering the list is cheap and eliminates the long-lived cursor entirely.

    Alternatives, if streaming must be preserved:

    • Add .noCursorTimeout(true) to the find() used by list()/findAll() so the server never reaps it (the reactive driver closes it on completion/cancel - but a cancelled/aborted stream risks leaking a cursor).
    • Increase batchSize large enough that a single batch covers the whole collection, so no getMore is ever issued - brittle and bounded by the 16MB batch limit.

    Recommendation: go with collectList() materialization for userDAO.list() and resourceDAO.findAll() inside CalendarEventsReindexService.reindex. It is the least surprising and fully removes the timeout window. Worth adding a regression test asserting the source listing is fully consumed before per-item work begins.

  • comment on #403 — @chibenwa, 2026-07-01 14:56

    Analysis: why a filter-less calendar-query REPORT times out while export is fine

    Context

    The failing side is the reindex task in twake-calendar-side-service (linagora/twake-calendar-side-service#887). It enumerates every event of a calendar via:

    calDavClient.calendarQueryReportXml(user.username(), calendarURL, CalendarQuery.ofFilters())

    CalendarQuery.ofFilters() is an empty filter — i.e. a CalDAV calendar-query REPORT with no time-range and no prop-filter, asking for all objects + calendar-data. The MongoCursorNotFoundException (error 43) reported on the James side is a downstream symptom: the caller keeps its own Mongo cursor (iterating users/calendars) open while it waits for Sabre to answer; Sabre is so slow on a big calendar that the idle cursor exceeds its server-side timeout and gets reaped. The real problem to fix is here in esn-sabre: the filter-less REPORT is O(N) round-trips.

    Root cause: N+1 in the vendored calendarQueryReport

    The XML REPORT is handled by Sabre\CalDAV\Plugin::calendarQueryReport() (vendor/sabre/dav/lib/CalDAV/Plugin.php:622):

    $nodePaths = $node->calendarQuery($report->filters);   // 1 query -> list of URIs

foreach ($nodePaths as $path) {
    list($properties) =
        $this->server->getPropertiesForPath($requestUri . '/' . $path, $report->properties);  // <-- one PROPFIND *per event*
    ...
}

    calendarQuery() runs a single Mongo query and returns just the URIs, but then the plugin fetches calendar-data one event at a time via getPropertiesForPath(). Each iteration triggers a separate getCalendarObject() → a separate Mongo lookup. For the user in #887 that is ~26 500 individual Mongo round-trips + 26 500 VObject parses in a single request. That is what blows past every reasonable timeout.

    Why export is fast

    The ICS export path (Sabre\CalDAV\ICSExportPlugin::generateResponse, vendor/.../ICSExportPlugin.php:210) has no filter, so it takes the fast branch:

    $nodes = $this->server->getPropertiesForPath($path, [$calDataProp], 1);  // single Depth:1 fetch

    A single Depth-1 propfind resolves to Calendar::getChildren() → Backend\Mongo::getCalendarObjects($calendarId), i.e. one Mongo query that returns every object with its data. No per-event round-trip, no N+1. Same data, one query instead of N.

    Note esn-sabre already learned this lesson for the JSON report: CalendarObjectService::calendarQueryWithAllData() / getMultipleDAVItemsOptimized() fetch URI + calendardata + etag in a single query ({uri: {$in: [...]}}, see CalendarObjectDAO::findByCalendarIdAndUris). The XML calendar-query REPORT never got that treatment and still goes through the stock per-path Sabre loop.

    Proposed fix

    ESN\CalDAV\Plugin already extends Sabre\CalDAV\Plugin (lib/CalDAV/Plugin.php:19), so we can override calendarQueryReport to batch the data retrieval instead of looping one path at a time:

    • Minimal / low-risk: replace the per-path getPropertiesForPath() loop with a single getPropertiesForMultiplePaths($nodePaths, $report->properties) (exactly what ICSExportPlugin does). This turns N getCalendarObject calls into one getMultipleCalendarObjects ($in query). Behaviour, filtering and output are unchanged.
    • Best: for the common empty-filter case (no post-filter needed — see buildQueryFromFilters, requirePostFilter=false), route straight to the existing calendarQueryWithAllData() so URIs + data come back in a single query and each object is parsed at most once. Fall back to the standard path only when a real post-filter is present.

    Either option removes the N+1 and brings the REPORT in line with export's single-query cost, which should make the reindex complete well within the caller's cursor timeout. On the twake side it would also be worth raising/refreshing the enumeration cursor (noCursorTimeout or smaller batches) as defence in depth, but that's secondary once the REPORT itself is fast.

    I can prepare the calendarQueryReport override with tests if you want to proceed.

    • https://mail-{workplaceFqdn}
