  • comment on #784 — @vttranlina, 2026-06-09 10:56

    The current nginx config does not define a default_server for HTTPS port 443.

    Because of that, any request that does not match a configured server_name, for example:

    https://<nginx-ip>/.env
https://<unknown-domain>/.git/config
https://<random-host>/favicon.ico

    may fall back to the first server block.

    In the current config, the first HTTPS server block is:

    server {
    listen 443 ssl;
    server_name twcalendar.linagora.com;

    location / {
        proxy_pass http://{publicIP}:9989;
    }
}

    As a result, those unrelated requests may be proxied to: http://{publicIP}:9989

    which is the calendar side service, and the application logs:

    FallbackProxy Not found GET /.env
FallbackProxy Not found GET /.git/config
FallbackProxy Not found GET /favicon.ico

    Proposed fix: Add bellow block at line 29

    server {
    listen 443 ssl default_server;
    server_name _;

    ssl_certificate /etc/letsencrypt/live/twcalendar.linagora.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/twcalendar.linagora.com/privkey.pem;

    return 444;
}

    After this change, HTTPS requests that do not match any known domain will be stopped by nginx and will no longer be proxied to calendar side service.

    // When nginx returns 444, it closes the client connection immediately without sending an HTTP response body or a normal HTTP status response.

    # User calendar management

User calendar management is done by proxying esn-sabre JSON API calls over webadmin !

## Listing calendars (users)

GET /users/btellier@linagora.com/calendars

WOULD be translated to:

GET http://sabre/calendars/5f50a663bdaffe002629099c.json?personal=true&sharedDelegationStatus=accepted&sharedPublicSubscription=true&withRights=true

Authenticated by impersonating the user.

WHERE 5f50a663bdaffe002629099c is the user id associated to btellier@linagora.com

## Listing calendars (resources)

GET /resources/5f6c478d9ff88b001f4a2106/calendars

WOULD be translated to:

GET http://sabre/calendars/5f6c478d9ff88b001f4a2106.json?sharedPublic=true

Authenticated with technical token of the domain

    Analysis

    This issue mirrors #711 (calendar CRD admin) but for address books / CardDAV. The request is to expose a webadmin HTTP interface so admins can manage a user's address books without going to Sabre directly.

    What already exists

    CardDavClient (calendar-dav/) already has the low-level methods needed:

    Method Sabre call
    listUserAddressBookIds(Username, OpenPaaSId) GET /addressbooks/{userId}.json?personal=true&shared=true&subscribed=true&…
    createUserAddressBook(Username, OpenPaaSId, id, name) POST /addressbooks/{userId}.json
    deleteUserAddressBook(Username, AddressBookURL) DELETE /addressbooks/{userId}/{bookId}.json
    exportAddressBook(Username, AddressBookURL, queryParams) GET /addressbooks/{userId}/{bookId}?export

    The calendar-side equivalent UserCalendarRoutes (calendar-webadmin/) is a clean template: it resolves the user from the DAO, proxies to Sabre impersonating that user, and maps errors to webadmin ErrorResponder halt exceptions.

    Proposed webadmin API

    Following the same pattern as UserCalendarRoutes (base path /users):

    # List address books
GET /users/{username}/addressbooks

→ GET http://sabre/addressbooks/{userId}.json?contactsCount=true&inviteStatus=2&personal=true&shared=true&subscribed=true
  Authenticated by impersonating the user.
  Response: raw JSON from Sabre (same approach as listCalendars in UserCalendarRoutes)

# Create address book
POST /users/{username}/addressbooks
Body: {"id":"<optional-uuid>","dav:name":"My Contacts","carddav:description":"<optional>"}

→ POST http://sabre/addressbooks/{userId}.json
  Authenticated by impersonating the user.
  Response: 201 Created with {"id":"<uuid>"}

# Delete address book
DELETE /users/{username}/addressbooks/{addressBookId}

→ DELETE http://sabre/addressbooks/{userId}/{addressBookId}.json
  Authenticated by impersonating the user.
  Response: 204 No Content

    Implementation plan

    1. CardDavClient — add a raw-bytes listing method

    listUserAddressBookIds returns a Flux<AddressBook> (id + type only). For the admin listing endpoint we want to forward the full Sabre JSON, just like findUserCalendarsAsBytes does for CalDav. Add:

    public Mono<byte[]> listUserAddressBooksAsBytes(Username username, OpenPaaSId userId) {
    String uri = "/addressbooks/%s.json?contactsCount=true&inviteStatus=2&personal=true&shared=true&subscribed=true"
        .formatted(userId.value());
    return httpClientWithImpersonation(username)
        .headers(h -> h.add(HttpHeaderNames.ACCEPT, "application/json"))
        .get().uri(uri)
        .responseSingle((response, buf) -> { /* 200 → asByteArray(), else error */ });
}

    2. CardDavClient — add address book existence check

    UserCalendarRoutes guards DELETE/PATCH with calendarExists(). The address book equivalent should check existence before deleting, or at minimum handle the 404 from Sabre gracefully. Add:

    public Mono<Boolean> addressBookExists(Username username, AddressBookURL url) { … }

    3. New UserAddressBookRoutes class

    Structure mirrors UserCalendarRoutes exactly:

    public static final String BASE_PATH = "/users";
// already shared with UserCalendarRoutes, no conflict

private static final String ADDRESSBOOKS_PATH = BASE_PATH + SEPARATOR + USERNAME_PARAM + SEPARATOR + "addressbooks";
private static final String ADDRESSBOOK_PATH = ADDRESSBOOKS_PATH + SEPARATOR + ADDRESSBOOK_ID_PARAM;

    Endpoints:

    • service.get(ADDRESSBOOKS_PATH, this::listAddressBooks)
    • service.post(ADDRESSBOOKS_PATH, this::createAddressBook)
    • service.delete(ADDRESSBOOK_PATH, this::deleteAddressBook)

    Both UserCalendarRoutes and UserAddressBookRoutes share /users as their base path — Spark routes are matched by full path, so there is no collision.

    4. Register in CalendarRoutesModule

    routesMultibinder.addBinding().to(UserAddressBookRoutes.class);

    5. Integration test UserAddressBookRoutesTest

    Same pattern as UserCalendarRoutesTest: uses SabreDavExtension + Docker Sabre, creates a user, exercises list/create/delete via REST-assured against a live WebAdminServer.

    Notable differences vs calendar admin (#711)

    Analysis

    This issue bundles three related features around booking-link traceability. Below is a walkthrough of how each maps onto the current code, with concrete touch points and a few design caveats worth deciding before implementation.

    1. Custom ICS field referencing the booking link

    Booking events are generated in BookingLinkEventIcsBuilder. It already stamps two custom properties on every VEVENT (X-PUBLICLY-CREATED=true, X-PUBLICLY-CREATOR=<email>) and, when relevant, X-OPENPAAS-VIDEOCONFERENCE (build / buildEvent, lines 87-135).

    Adding a reference is straightforward: introduce e.g. X-OPENPAAS-BOOKING-LINK and add it as an XProperty in buildEvent. The blocker is that build(...) does not currently receive the BookingLink — only the BookingRequest, organizer and duration. The public id lives on the BookingLink in BookingLinkReservationService.createBooking (lines 96-104), so the call needs to be threaded through (pass bookingLink.publicId() into build).

    Caveat — which identifier to embed. BookingLinkDAO.resetPublicId(...) mutates the publicId field on the same Mongo document (MongoDBBookingLinkDAO line 162), and reset exists precisely for the compromise scenario this issue targets. If events embed the public id and an admin then resets it, every previously created event becomes unreachable by a "cleanup by current public id" search. Options:

    • Embed the public id and make the cleanup task accept an explicit id (so the admin can run it against the old id before/after reset). Simplest, but the operator must know the leaked id.
    • Embed a stable internal identifier (the Mongo _id, not currently exposed on the BookingLink record) so cleanup survives a reset. More robust, but requires surfacing a stable id on the model/DAO.

    I would decide this explicitly; my preference is a stable internal id since the whole point is recovering from a compromised (and likely reset) link.

    2. Index the booking link + expose a search criterion

    • The live indexing path goes through AMQP: EventFieldConverter.from(...) switches over jCal EventProperty names (lines 92-138). X-OPENPAAS-VIDEOCONFERENCE is wired here as EventProperty.VIDEOCONFERENCE (EventProperty line 69). A new x-openpaas-booking-link constant + case is needed.

    • Add it to CalendarEventsDocument (fromEventFields / toEventFields) and a CalendarFields constant + an indexed KeywordProperty in CalendarEventIndexMappingFactory.createTypeMapping (lines 188-211).     3. User task to clean up (delete) all events from a booking link

    The webadmin task framework already has the right shape: see CalendarEventsReindexTask and the registrations in CalendarRoutes/CalendarRoutesModule, plus the deletion-step pattern in CalendarSearchDeletionTaskStep. The booking-link admin surface to hang it off already exists in BookingLinkUserRoutes.

    A cleanup task needs to, for a given booking link:

    1. Find the matching events — naturally implemented on top of feature #2 (search by booking-link id, scoped to the owner's calendar).
    2. Delete from the source of truth (CalDAV), not just the index. This is the key point: OpensearchCalendarSearchService.delete/deleteAll only touch OpenSearch.     3. Be idempotent and report counts (processed/deleted/failed) like the other tasks.

    Caveat: search results are capped (EventSearchQuery.MAX_LIMIT = 256, and source calendars capped at 256 in OpensearchCalendarSearchService). A cleanup task must paginate/loop rather than rely on a single search call, otherwise links with many bookings will not be fully cleaned.

    Feature #1 (emit the X-prop) and #2 (index + search) are prerequisites for a search-based #3. I would implement: (a) emit + index + parse the new field with the identifier decision above, (b) the search criterion + REST exposure, (c) the cleanup task built on the search, deleting via CalDAV.     Analysis

    Root cause

    The reindex-calendar-events task iterates users and resources by streaming an open MongoDB cursor while doing very slow per-item work on it, and the cursor is reaped server-side before the next getMore.

    In CalendarEventsReindexService.reindex(...) the source is:

    Flux.concat(
    userDAO.list().concatMap(user -> collectEvents(...)),          // one open cursor
    resourceDAO.findAll().concatMap(resource -> collectEvents(...))) // another open cursor

    Both MongoDBOpenPaaSUserDAO.list() and MongoDBResourceDAO.findAll() are plain streamed cursors with no batching/timeout tuning:

    • MongoDBOpenPaaSUserDAO.java:158-160 -> Flux.from(database.getCollection(COLLECTION).find())
    • MongoDBResourceDAO.java:96-97 -> Flux.from(collection.find()).map(...)

    The reactive driver (com.mongodb.reactivestreams) reads this cursor batch by batch: it fetches one batch, emits it downstream, and only issues the next getMore once that batch is drained. Draining a batch here is extremely expensive and rate-limited:

    • default calendarsConcurrency = 1 (CalendarEventsReindexTask.java:43) -> users and their calendars are processed sequentially via concatMap;
    • for each item we deleteAll from the search index, hit CalDAV (calendarQueryReportXml), parse ICS, and reindex;
    • the whole indexing pipeline is throttled to eventsPerSecond (ReactorUtils.throttle(), lines 142-145).

    With 26540 events and this sequential/throttled processing, the time spent between two consecutive getMore calls on the user cursor easily exceeds MongoDB's default idle-cursor timeout (cursorTimeoutMillis, 10 minutes). The server then reaps the cursor, and the next getMore fails - exactly the stack trace reported:

    MongoCursorNotFoundException: cursor id ... not found (code 43, CursorNotFound)
  at ...AsyncCommandBatchCursor.lambda$getMoreLoop$2

    This propagates up past the per-user/per-resource onErrorResume handlers (those wrap the inner collectEvents monos, not the outer source Flux), reaches the top-level .onErrorResume at lines 155-158, which logs "Task reindex is incomplete" and returns Task.Result.PARTIAL. That is why the task ends failed/partial even though failedEventCount = 0: the failure is the source cursor dying, not any individual event.

    Fix

    Decouple the slow downstream processing from the live driver cursor. The simplest, robust option is to drain the user/resource listing into memory first, closing the cursor quickly, then process:

    userDAO.list()
    .collectList()
    .flatMapMany(Flux::fromIterable)
    .concatMap(user -> collectEvents(context, user, runningOptions.calendarsConcurrency()))

    (and the same for resourceDAO.findAll()). OpenPaaSUser/Resource are lightweight records and their cardinality is bounded (thousands, not the 26k events), so buffering the list is cheap and eliminates the long-lived cursor entirely.

    Alternatives, if streaming must be preserved:

    • Add .noCursorTimeout(true) to the find() used by list()/findAll() so the server never reaps it (the reactive driver closes it on completion/cancel - but a cancelled/aborted stream risks leaking a cursor).
    • Increase batchSize large enough that a single batch covers the whole collection, so no getMore is ever issued - brittle and bounded by the 16MB batch limit.

    Recommendation: go with collectList() materialization for userDAO.list() and resourceDAO.findAll() inside CalendarEventsReindexService.reindex. It is the least surprising and fully removes the timeout window. Worth adding a regression test asserting the source listing is fully consumed before per-item work begins.

    Analysis

    I confirm the race condition is real, and I want to add some precision on why the sequence guard is bypassed, plus flag a second problem that makes the "obvious" fixes insufficient.

    1. Why the delete bypasses the sequence check

    The per-document sequence guard lives in OpensearchCalendarSearchService#indexWithSequence, via the painless script UPSERT_WITH_SEQUENCE_SCRIPT:

    if (ctx._source.sequence == null || params.sequence > ctx._source.sequence) {
    ctx._source.putAll(params.doc);
}

    The request is built with .scriptedUpsert(false) and .upsert(docMap). With scriptedUpsert(false), the script only runs when the document already exists. When the document is missing, OpenSearch inserts the upsert doc verbatim, without running the script.

    So in the recurring path (CalendarEventHandler#indexEvents):

    1. delete(calendarURL, eventUid) removes every occurrence document for the eventUid.
    2. index(calendarEvents) re-indexes each EventFields — but since the documents were just deleted, every upsert takes the insert-verbatim branch and the sequence > ctx._source.sequence check never executes.

    Result: the sequence guard is fully disabled for recurring events with overridden occurrences, exactly as the issue describes. The onErrorResume(isVersionConflict) in indexEvents does not help either — after a delete there is no existing document, so there is no version conflict to catch.

    2. It is not only reordering — it is also non-atomic + concurrent

    Two aggravating factors beyond the A-then-B reorder example:

    • The consumer processes messages with flatMap(..., DEFAULT_CONCURRENCY) (see doConsumeCalendarEventMessages), so messages for the same eventUid run concurrently.
    • delete().then(index()) is not atomic. Even with in-order delivery, an interleaving like A.delete -> B.delete -> B.index(11) -> A.index(10) (or worse, one that drops documents entirely) is possible.

    For the single-event path this is tolerable because the sequence-guarded upsert plus retryOnConflict(3) converges to the newest sequence. The recurring path has neither protection.

    3. The subtle part: naive per-document guards still resurrect deleted occurrences

    It is tempting to "just guard the delete too" (e.g. delete-by-query with sequence < incoming, then sequence-guarded upsert). That fixes the overwrite case but not the removed-occurrence case:

    • Old state (seq 10) has an overridden occurrence X. New state (seq 11) removes that override.
    • B(11) applies: the X document is gone. Good.
    • Stale A(10) arrives later, re-indexes its occurrence set which still contains X. Since the X document no longer exists, the upsert takes the verbatim-insert branch and resurrects the deleted occurrence.

    The root cause is structural: with one OpenSearch document per occurrence, a missing document is indistinguishable between "new" and "deleted by a newer message", so no per-document guard can reject a stale insert.

    Options

    • (A) Single document per recurring event (recommended for correctness). Store the whole CalendarEvents as one document keyed by eventUid+calendarURL, with occurrences as nested fields for search (inner-hits on query). A single atomic sequence-guarded upsert (reusing UPSERT_WITH_SEQUENCE_SCRIPT with the master/max sequence) then makes the whole update all-or-nothing — no delete, no resurrection, no reorder window. This is the design the issue questions, but it is precisely what makes the guard atomic. Cost: mapping change + search() must project inner-hits back to EventFields.

    • (B) Version-guarded delete + guarded upsert. Cheaper, but only a partial fix: it handles the overwrite race (sections 1/2) yet still suffers the resurrection problem (section 3). Being fully correct would need an additional tombstone / event-version record, which starts to approximate (A) anyway.

    • (C) Serialize per eventUid (groupBy(eventUid).concatMap(...)) to remove intra-instance interleaving. Necessary but not sufficient: it does not reorder by sequence and does not cover multi-instance deployments, so it must still be combined with a version guard.

    My recommendation is (A): it is the only option that closes all three problems (overwrite, non-atomicity, resurrection) without bolting on extra tombstone bookkeeping. If the search-side nested-mapping change is too costly short-term, (B)+(C) narrows the window but should be documented as a mitigation rather than a complete fix.

    Short answer: yes, the slot is validated server-side — a hand-crafted curl cannot book an arbitrary date. But there are a couple of gaps worth deciding on.

    Where validation happens

    POST /api/booking-links/{id}/book → BookingLinkReservationService.book() calls validateSlotAvailability() before creating the event (BookingLinkReservationService.java:79-95):

    return bookingLinkSlotsService.computeSlots(bookingLink, queryStart, queryEnd)
    .filter(slots -> slots.contains(new AvailabilitySlot(startUtc, bookingLink.duration())))
    .switchIfEmpty(Mono.error(() -> BookingLinkReservationException.notAvailable(...)))

    The requested startUtc must be exactly equal to one of the server-computed AvailabilitySlots (same start, same duration). Those slots are recomputed on the server from the same source as the public /slots endpoint (BookingLinkSlotsService.computeSlots): availability rules minus busy intervals from CalDAV (AvailableSlotsCalculator.Default). An unavailable/misaligned slot yields 422 UNPROCESSABLE_ENTITY. So:

    • Times that do not fall on the availability grid are rejected — slots are generated minute-aligned, stepping by duration from the availability-range start (AvailableSlotsCalculator.java:99-108). A startUtc with stray seconds or off-grid minutes will not match.
    • Times overlapping an existing busy event are rejected (unavailableTimeRanges removed in computeAvailabilityRanges).
    • Times outside the configured weekly/fixed availability rules are rejected.

    The fear of full "openbar" registration is therefore not the case for a link that has availability rules configured.

    Gaps worth considering

    1. No lower time bound / no minimum-notice check. Nothing checks that startUtc is in the future. Weekly rules are recurring, and the validation window is derived purely from the user-supplied startUtc (queryStart = startUtc - duration, queryEnd = startUtc + duration), independent of "now". A curl caller can therefore book a slot in the past as long as it matches a weekly availability rule (e.g. last Tuesday 10:00). The Clock injected into the service is only used for the ICS DTSTAMP, not for validation. Note the public /slots endpoint (BookingLinkSlotsRoute) also never clamps from to now, so the UI could equally query past ranges. A configurable "minimum booking notice" / future-only guard would close this.

    2. Empty availability rules means effectively open. When bookingLink.availabilityRules() is empty, BookingLinkSlotsService.toComputeSlotsRequest falls back to a FixedAvailabilityRule spanning the entire query window (BookingLinkSlotsService.java:80-85). Combined with gap 1, that means any minute-aligned, non-busy instant becomes bookable at any date. If a link is never meant to exist without rules, that invariant should be enforced at creation; otherwise this is the real "openbar" scenario.

    3. No throttling on the public endpoint. Independent of slot validity, /book is unauthenticated (PublicRoute); nothing rate-limits creation. Even valid-slot bookings can be spammed (one event per free slot). Rate-limiting / captcha is a separate hardening item if abuse is a concern.

    Suggestion

    The core slot-availability validation is present and correct. I would recommend: (a) add a future/minimum-notice check in validateSlotAvailability using the injected Clock, and (b) decide whether a booking link may exist without availability rules, enforcing it at creation if not.

    Current behaviour

    The booking endpoint POST /api/booking-links/{publicId}/book is served by BookingLinkReservationRoute, which extends PublicRoute — i.e. it is fully unauthenticated (calendar-rest-api/.../routes/PublicRoute.java). The booker's identity is taken verbatim from the request body:

    We cannot require auth unconditionally: external/anonymous booking is the whole point of a booking link (the frontend excal form). So the check has to be conditional — enforce identity only when the declared identity belongs to a registered user, and leave genuinely external bookers untouched. Hence "when possible".

    Proposed approach

    1. Make the route optionally authenticated. Today the split is binary: PublicRoute (no auth) vs CalendarRoute (authenticator.authenticate(req), which hard-fails on missing/invalid credentials). We need a third mode: attempt authentication, tolerate its absence, but still reject genuinely invalid credentials. The frontend (#1109) will forward OIDC creds when present and fall back to the anonymous excal form otherwise, so the backend must accept both a valid session and no session.

    2. Detect a registered identity. Resolve creator.email via OpenPaaSUserDAO.retrieve(Username.of(email)). Non-empty means the declared creator is a local user.

    3. Enforce the match. If the creator is a registered user:

      • require an authenticated session, and
      • require session.getUser() to match creator.email (case-insensitive),
      • otherwise reject with 401 (no/invalid creds) or 403 (authenticated as someone else).

      If the creator is not a registered user, keep the current anonymous flow unchanged.

    4. Trust the platform identity, not the body. When authenticated, derive the creator's display name (and ideally email) from the resolved OpenPaaSUser.fullName() rather than the client-supplied name, as the issue states ("the registered user display name is then used"). This also prevents a valid user from setting a misleading CN.

    Points to settle before implementing

    • Identity resolution must match auth's view of identity. Detection in step 2 must use the same username/alias resolution as the authenticator (see UserNameResolver), otherwise an attacker could dodge the check by using an alias of a registered user that retrieve() does not resolve but which is still a real inbox. This is the main correctness risk.
    • Additional attendees. additional_attendees are still fully self-declared. Impersonating them is lower risk (equivalent to inviting someone to any event), so I would scope this change to creator only — but worth noting explicitly.
    • Invalid-credentials policy. Decide whether presenting invalid/expired credentials on an anonymous booking should fail hard or degrade to anonymous. Failing hard is safer and avoids a silent downgrade.
    • CORS / preflight are already handled by the route builder; no change needed there.
    • Tests. Add cases to BookingLinkReservationRouteTest: registered creator without auth => rejected; registered creator with mismatched auth => rejected; registered creator with matching auth => created and display name overridden; external creator => unchanged.

    Recommendation

    Worth doing — it closes a real spoofing hole at low cost and improves UX for logged-in users. The only non-trivial part is wiring optional authentication into a route that is currently public, and making sure registered-user detection uses the exact same identity/alias resolution as authentication. I would implement it as: optional auth in the route + a check in BookingLinkReservationService.book(...) that takes an Optional<Username> authenticatedUser and enforces the rules above.

  • inline comment on PR #811 — @vttranlina, 2026-06-18 13:06

    retryOnConflict here is not meant to bypass the SEQUENCE rule.

    It handles OpenSearch optimistic concurrency conflicts caused by concurrent consumer processing.

    Example:

    Initial state:

    • Current document SEQUENCE = 9

    Concurrent messages:

    • Message A: SEQUENCE = 10
    • Message B: SEQUENCE = 11

    Race:

    • A commits first, document becomes SEQUENCE = 10
    • B then fails with an OpenSearch version conflict, even though B is newer

    With retryOnConflict:

    • B retries against the latest document state: SEQUENCE = 10
    • The same script runs again
    • Since 11 > 10, B is applied
