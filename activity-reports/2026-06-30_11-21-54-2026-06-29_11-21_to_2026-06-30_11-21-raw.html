  • comment on #1093 — @chibenwa, 2026-06-29 17:08

    Analysis — x-openpaas-videoconference jCal type retro-compat

    Context: linagora/twake-calendar-integration-tests#280 flips the jCal value type of the x-openpaas-videoconference property from unknown to uri (e.g. ["x-openpaas-videoconference",{},"uri",""]). The backend now normalizes stored events to uri (PUT bodies sending unknown still return 201, and the subsequent GET returns uri). So the frontend must keep working whether the property comes back as unknown, uri, or anything else.

    1. Read / parse path — already format-agnostic ✅

    parseEventProperty (common/src/features/Events/utils/parseCalendarEvent.ts:232) destructures the jCal tuple as const [key, params, , value] = prop — the type element (3rd position) is discarded. The x-openpaas-videoconference handler (:170) only reads value. So unknown, uri, text, etc. all parse identically.

    This is already proven by the existing unit test eventUtils.test.ts:38, which feeds type text and passes. No change required to read both formats.

    2. Write path — hardcodes unknown, still functional ⚠️

    makeVevent (common/src/features/Events/utils/makeVevent.ts:29-33) always emits:

    ['x-openpaas-videoconference', {}, 'unknown', event.x_openpass_videoconference ?? null]

    This is the only 'unknown' literal in the events feature. The backend accepts it (integration tests PUT unknown → 201), so nothing breaks. To match the backend's canonical form it would be cleaner to emit 'uri' and an empty string instead of null, but it is not strictly required for correctness.

    3. Real retro-compat risk — series metadata diff in makeSeriesJCal ⚠️

    detectChangedMetadataFields (common/src/features/Events/transformers/makeSeriesJCal.ts:50-66) compares old vs new master properties with a full JSON.stringify of the whole tuple, including the type element. x-openpaas-videoconference is in METADATA_FIELDS (:19).

    After the backend change, the old master loaded from the server serializes as [...,"uri",""], while the new master built by makeVevent serializes as [...,"unknown",null]. The mismatch is twofold (type uri≠unknown, value ""≠null), so the field is flagged as "changed" on every series edit even when the video link was untouched — triggering needless propagation to all override instances and sequence bumps. (The "" vs null value mismatch likely pre-dates this type change; the type flip just widens the gap.) The existing test EventTransformers.test.tsx:357 only exercises the genuine-change case, so it would not catch this false positive.

    Recommendation

    • Read: nothing to do — type is already ignored. Optionally add a regression test parsing the property with type uri to lock the behavior in.
    • Write: align makeVevent to emit uri + empty string for the canonical form.
    • Series: make the metadata-diff in makeSeriesJCal robust to the type/empty-value representation — either compare only name/params/value (dropping the type element) or normalize both sides — so that a no-op video-link field stops being detected as a change. Add/adjust a test covering "master loaded as uri, edited via unknown writer → no spurious override propagation".

    Net: both formats are already supported on the read path; the concrete work is on the write side (canonical uri) and hardening the series metadata comparison.
