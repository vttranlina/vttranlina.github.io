Highlights

Backend / Side Service / Sabre

Booking links

  • WebAdmin Booking Link CRUD was added for frontend/admin usage in pull/839 / issues/837.
  • Booking links gained optional name and description in pull/861, with a MongoDB persistence bug caught by tests and fixed during review.
  • Auto-accept support was added in pull/863; the review clarified that booker notification on organizer acceptance is handled by regular IMIP.
  • Generated booking events now reference the booking link through ICS metadata in pull/865, with follow-up indexing/search support in pull/869.
  • Admin cleanup for all events created by a booking link was implemented in pull/876. A review caught an unstable paging-while-deleting bug; the implementation was changed to use CalDAV listing as the source of truth instead of paged OpenSearch results.
  • Public booking slots now expose owner information and booking-link metadata (name, description, autoAccept) via pull/878, and support requested timezone output via pull/880.
  • Booking confirmation flow gained a token signer, reservation response token, event get route, and cancel route via pull/873, pull/874, pull/886, and pull/881. Review clarified token naming and cancel idempotency/error handling.
  • Booking request acknowledgement emails were added and then enriched with booking-link details in pull/852 and pull/906. Review explicitly avoided unauthenticated booker-provided text in sensitive emails.
  • Server-side booking hardening was improved: past slots/booking/cancellation are rejected in pull/915, and availability is enforced even when no availability rules are configured in pull/916.
  • A new open issue tracks validating participant identity when the declared booker is a registered user, to prevent spoofing through the unauthenticated booking endpoint (issues/919). The current proposal is a configurable flag defaulting to current behavior.

Address book / CardDAV webadmin

  • User calendar WebAdmin management routes were implemented in pull/789, proxying Sabre APIs for list/create/delete/public visibility/invitee operations.
  • Address book admin WebAdmin routes were added in pull/851 / issues/801. The design includes list/create/delete, public visibility, and delegation/sharee management; review added API documentation and fixed system address book deletion to return a clear 400 instead of 500.
  • Domain settings patch routes and public calendar visibility domain setting were added in pull/783 and pull/793. Review clarified “absent = unchanged” vs “present null = reset” semantics.

Team Calendar

  • Grooming on issues/515 clarified key decisions: Team Calendar has a side-service-managed entity, DAV permissions manage members, Team Calendars are event holders rather than participants, and People Search must expose them for calendar quick search.
  • Entity repository API and memory implementation landed in pull/857, MongoDB implementation in pull/858, and WebAdmin entity CRUD in pull/901.
  • Review on pull/901 clarified create/delete behavior: create is not idempotent; delete should be idempotent and covered by tests.
  • Team Calendar list notifications are ignored by AMQP consumers in pull/902, avoiding incorrect handling as regular calendar notifications.
  • Team Calendar People Search support was merged in pull/917. Review noted that the current approach is fine for smaller domains but may need event-bus + OpenSearch-backed providers for very large domains.
  • Open Team Calendar follow-ups remain: members WebAdmin CRUD (issues/832), integration tests (issues/833), demo video (issues/835), and renaming DAV collection when a Team Calendar is renamed (issues/859).

Search, reindex, AMQP, and production stability

  • Shared/delegated calendar search was implemented through DAV source resolution and OpenSearch indexing updates in pull/788, pull/794, and pull/806. A resource-calendar indexing gap was identified and handled.
  • Reindex moved from export to REPORT API and fixed resource .ics naming in pull/791 and pull/814.
  • Large reindex-calendar-events tasks hit MongoCursorNotFoundException; root cause was long-lived Mongo cursors while doing slow downstream work. pull/891 materializes user/resource lists first to close cursors early. This was tied to release/deployment tracking in issues/871.
  • Recurring-event indexing race around overridden occurrences was deeply analyzed and fixed in pull/898. The final direction keeps occurrence documents searchable, uses OpenSearch collapse on UID for representative results, prunes removed occurrences by identity/recurrenceId only when the incoming master update wins, preserves recurrenceId, and documents upgrade/reindex instructions.
  • Malformed iTIP recipient/sender handling was improved in pull/912: malformed local-delivery addresses are sanitized where possible; invalid addresses are ignored/acked instead of dead-lettered, while valid recipients continue to be processed.
  • A proposed parser workaround for observance-less VTIMEZONE in pull/897 was rejected as too hacky because affected records were test data; issues/896 was closed with “ignore” rather than sanitizer.
  • AMQP shutdown is more graceful via pull/847, and OpenSearch concurrent updates now retry conflicts while respecting SEQUENCE guards via pull/811.
  • Audit logging for Sabre CRUD operations is open in pull/903. Review forced a safer design: one audit queue instead of 22, no raw event/vCard body logging, human-readable messages, and richer audit context. It still needs improvement to include who performed the action and on what target.
