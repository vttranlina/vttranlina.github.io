Validation / RFC behavior

  • Organizer PUT validation was designed in issues/389 and merged in pull/395. Decisions captured there: VEVENT-only, opt-in first, reject ATTENDEE without ORGANIZER, and keep ORGANIZER consistent across VEVENT overrides.
  • A production-facing regression appeared after enabling organizer validation: attendees could not update PARTSTAT because the plugin rejected unchanged organizer data. The issue was tracked in issues/408 and fixed in pull/410.
  • CalendarObjectValidator was reviewed in issues/398. Key caveat: validation is opt-in via Prefer: handling=strict, so frontend flows must send the header or the validator gives no protection.
  • pull/399 tightened validation by requiring DTSTART for RRULE/EXDATE and documenting stricter TZID matching.

CalDAV REPORT performance and ACL

  • issues/403 identified the XML calendar-query REPORT N+1 pattern as the likely root cause of reindex timeouts: large calendars caused thousands of Mongo round-trips and VObject parses.
  • pull/406 implemented batched and single-query report paths. Review requested moving report logic into a dedicated class; this was done before merge.
  • Preprod testing reported REPORT time around 3–4s versus 12s for the same query against prod, but a reindex run still failed after processing 151,479 events with 4 failures. This needs follow-up before considering the path fully stable.
  • ACL behavior around optimized reports was adjusted after integration-test feedback in pull/406, with related test updates referenced in integration-tests pull/287 and integration-tests pull/288.
