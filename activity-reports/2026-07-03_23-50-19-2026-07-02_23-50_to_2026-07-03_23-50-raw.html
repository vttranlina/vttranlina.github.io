  • comment on #898 — @chibenwa, 2026-07-03 14:12

    Thanks @chibenwa — I looked into the raised concern.

    On max(SEQUENCE): the fix does not gate indexing on a max(SEQUENCE) of the whole recurring event. Each occurrence (master and every overridden instance) is upserted as its own sequence-guarded document, each carrying its own SEQUENCE, so per-instance sequences are respected independently. max(sequence) is used only as a lower bound to prune occurrences that dropped out of a payload, and never deletes a document we just wrote — so it cannot downgrade a newer per-instance state.

    On searching each VEVENT instance (the "Bob" example): this already works. Each occurrence is indexed as a separate document; the collapse on eventUid only groups documents that matched the query. When a keyword matches only an overridden occurrence (e.g. Bob invited on one instance but absent from the master), the master is not a hit, so the collapse group contains only the override and that override VEVENT is returned. keepMaster collapses to the master only when the master itself is also a match.

    This was already exercised indirectly by removedRecurrenceOccurrenceShouldNotRemainSearchableAfterUpdate (searching a keyword unique to an override returns that override). To pin your exact example I added a dedicated contract test, searchShouldReturnOverriddenOccurrenceWhenOnlyItMatches: a recurring master without Bob + one override with Bob as attendee → searching Bob returns the override occurrence. It runs against both the Memory and OpenSearch implementations.

    Pushed as 2cd862bb. Note: the OpenSearch integration test requires Docker and could not run locally; it relies on CI.
